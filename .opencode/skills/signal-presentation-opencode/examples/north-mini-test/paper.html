Measuring Acceptance Reliability in Coding-Agent CI Repair

Five-model breadth evidence and a 391-attempt depth study across 33 hidden-acceptance scenarios

Anil Yamangil

Independent Research · June 2026 · Canonical Technical Report · research suite

Home | Paper | Harness | Catalog | Evidence | Evaluator

Abstract

We present a two-stage empirical study investigating the "trust gap" in coding-agent CI repair — the difference between visible test success (public green) and actual correctness under hidden acceptance tests. In Stage A (Breadth), we evaluate five model families (Cohere North Mini, Google Gemma 4, Poolside Laguna XS.2, DeepSeek V4 Pro) at Pass@1. In Stage B (Depth), we conduct a head-to-head Pass@3 evaluation involving 391 retained attempts across 33 scenarios and two deployed agent systems. We demonstrate that the trust gap is heavily driven by task pack and prompt specificity rather than model identity. While sparse prompts on product workflows produce massive attempt-level false-green rates (43.3% combined), contract-visible prompts collapse this rate to 8.9% combined. For Laguna XS.2 / OpenRouter, the attempt-level false-green rate fell to 4.0% across contract-visible cells, and no maintenance or product scenario remained unsolved under any-of-three attempts. Retries do not monotonically improve outcomes: independent resampling occasionally regressed, replacing correct solutions with incorrect ones. We analyze stable attractors, tool-call distributions, and implementation limitations to outline engineering guidelines for coding-agent deployment.

Keywords: coding agents, CI repair, trust gap, false-green, hidden acceptance, Pass@3, evaluation harness, contract-visible

1. Plain-Language Summary

When an AI coding assistant completes a software bug fix and gets your continuous integration (CI) tests to pass, how confident can you be that the underlying bug is actually fixed? This report shows that public test suites are often too sparse to define correctness. AI agents easily optimize for whatever visible assertions they can see, producing a green CI check while violating hidden requirements (a false-green). We find that this "trust gap" is structural: it is present across all evaluated models, ranging from local 12B models to hosted frontier engines. However, exposing explicit contracts directly in the prompt (contract-visible) collapses false-greens on mechanical and refactoring tasks. For complex business logic, explicit contracts recover most scenarios, but per-attempt reliability remains weaker, meaning human verification or hidden acceptance gates remain essential.

2. Research Question and Terminology

Standard benchmarks evaluate whether an agent can produce code that passes unit tests. Our study audits the correctness of those fixes when they satisfy visible assertions. We define four core metrics to examine this reliability:

  • Attempt-level hidden pass rate: The percentage of all completed attempts that pass hidden acceptance tests.
  • Attempt-level false-green rate: The number of attempts that passed visible tests but failed hidden acceptance, divided by all attempts that passed visible tests.
  • Any-of-3 scenario success: The proportion of scenarios where at least one of three independent attempts successfully passed hidden acceptance.
  • All-of-3 scenario reliability: The proportion of scenarios where every single completed attempt passed hidden acceptance, indicating high strategy stability.

3. Evaluation System

The evaluation harness automates a 10-step pipeline: (1) clone a template repository to a fresh worktree, (2) run baseline tests to verify the starting red CI state, (3) execute the coding agent via the OpenCode CLI, (4) evaluate public tests to check visible success, (5) inject hidden acceptance tests, (6) execute hidden tests, (7) calculate classification outcomes, (8) extract the patch diff, (9) compute SHA256 hashes for all workspace states, and (10) persist results to a central SQLite database. Each attempt is run in an isolated worktree with no memory of prior trials. Baseline test results are recorded but not enforced. Historical patches are extracted via git diff and may omit untracked files. Integrity hashes are computed after the run is complete, not compared against real-time digests.

7. Attempt-Level Reliability

The attempt-level metrics reflect the chance that any individual run produces a correct patch. Under sparse conditions, attempt-level false-green rates are consistently high across both models (ranging from 24% to 80%). The contract-visible treatment lowers this rate substantially, but residual errors remain. For instance, in product workflows under contract-visible mode, Laguna's attempt-level false-green rate dropped to 6.1%, whereas North Mini's stayed at 32.3% (partly due to persistent failure attractors in support_sla_business_hours and billing scenarios).

8. Scenario Coverage and Stability

Any-of-three scenario success treats a scenario as solved if at least one attempt passes hidden tests. Laguna XS.2 reached 33/33 any-of-three success under contract-visible conditions, and North Mini reached 32/33 (failing to solve support_sla_business_hours). However, this any-of-three metric obscures instability. The all-of-three metric reveals how consistently the agent reproduces a fix: Laguna fell to 29/33 all-of-three reliability in CV, and North Mini reached 21/30 (retained matching cells), showing that even with clear specifications, agents fluctuate between correct and incorrect strategies.

9. Independent Retries Are Not Monotonically Better

A cumulative Pass@k metric cannot mathematically regress, but individual attempts can. Table 4 demonstrates that resampling independently from a model's generation distribution does not guarantee improvement and can result in regression.

In attempt 2, North Mini failed on env_bool_parser (which it solved in attempt 1) without gaining any new solved scenarios. This shows independent resampling occasionally recovered a failure, occasionally replaced a successful strategy with a failing one, and repeatedly reproduced stable errors.

10. Effect of Contract Exposure

Exposing the contract in the prompt lane collapses the false-green rate across both systems. On mechanical tasks (ci_forensics) and refactoring tasks (maintenance_value), Laguna’s attempt-level false-green rate fell to 5.6% and 0.0% respectively. North Mini showed similar reductions (to 5.6% and 4.0%). The spec is the primary lever of control: informing the agent of the hidden validation criteria redirects its search space toward true acceptance.

11. Failure Case Analysis

Certain scenarios present stable failure attractors. In `decimal_money_rounding`, Laguna generated byte-identical wrong patches on attempts 1 and 2, converting variables to `Decimal` but leaving `int(total * 100)` in place, which truncates half-cents instead of rounding. The model satisfies the visible assertions but fails the hidden half-cent test. The evaluator agent (DeepSeek V4 Pro) confirmed that the model optimistically declared success because its code satisfied the visible test suite. We conclude that when visible tests are sparse, agents converge to stable, incomplete attractors.

12. Operational Behavior and Gateway Limitations

North Mini was routed through two gateways: OpenCode Zen (for forensics and maintenance) and OpenRouter (for product workflows). We cannot separate gateway route effects from scenario pack differences. Because the OpenCode Zen gateway enforces strict daily quotas, 5 attempts are missing from the North Mini maintenance CV cells. Operational limitations, such as rate limits and quota walls, are significant variables in agent deployment.

13. Exploratory Trace and Patch Statistics

We analyze the relationship between run metrics and outcomes. These statistics are exploratory and should be interpreted with caution. Tool-call metrics are parsed from the logged stdout and may vary across runtime environments. Diff statistics only represent files tracked in the git diff; untracked files added by the model are omitted.

  • Tracked-file diff statistics: False-greens are slightly larger on average (7.4–7.7 changed lines vs 6.0–6.1 for passes), showing that models write more code to satisfy visible tests without matching hidden logic.
  • Tool-call counts: In the pooled retained traces, false-green attempts used fewer logged tool calls on average (10.7–10.8 vs 13.1–14.2). This association may reflect pack, lane, and scenario mix; a within-cell analysis is needed before interpreting it as an agent stopping mechanism.

14. Threats to Validity

  • Benchmark author bias: Hidden tests, templates, and baseline suites were written by the same team, which may introduce systematic omissions.
  • Incomplete evaluator coverage: Only 10 of 101 false-green attempts have been audited by the evaluator model. 91 false-greens remain unreviewed.
  • Inference differences: OpenRouter and OpenCode Zen gateways may serve different model versions or use different temperature configurations.
  • Code Audit Findings: The baseline test is recorded but not enforced. Incomplete runs skipped by matrix resume are assumed complete in SQLite. Evaluator fallback reviews are not currently filtered out from validation statistics.
