Pass@3 Trust Gap: Two Coding Agents, 33 Scenarios, and a Wall They Cannot Climb

Anil Yamangil

Independent Research · June 2026 · v2 — pass@3 matrix update · research suite

Home | Paper | Harness | Catalog | Evidence | Evaluator

Abstract

We run a head-to-head pass@3 evaluation of two independently trained 3B-active MoE coding agents — Cohere North Mini and Poolside Laguna XS.2 — across 33 CI-repair scenarios with hidden acceptance tests injected after the agent exits. On ci_forensics sparse, both models converge to identical 75% hidden pass rates (9/12 scenarios) at best-of-3, with two scenarios unconquered by both models (0 hidden passes across all 6 attempts each). Contract-visible prompts collapse the false-green rate from 28–31% to 5.6% for both models. Laguna reaches 100% hidden pass on maintenance_value with contract-visible prompts — zero false-greens across 30 runs. Product workflows sparse produces a 72.7% false-green rate with 7 of 11 scenarios completely unconquered at pass@3. Retries do not converge: North Mini regressed from pass@1 to pass@2, losing a solved scenario without gaining any. Both models hit first-output timeout walls in different cells. The evidence supports delegating mechanical and refactoring tasks with explicit specs while requiring hidden acceptance gates for business logic. The model choice matters less than the task type.

Keywords: coding agents, CI repair, trust gap, false-green, hidden acceptance, pass@3, evaluation harness, contract-visible

1. Introduction

Standard coding benchmarks measure whether a model can generate code that passes tests. This evaluation asks a different question: when a coding agent makes CI green, how often is the bug actually fixed? Observed 239 non-timeout runs across 2 models, 3 packs, 2 prompt modes, 3 attempts each

The distinction matters because visible tests create an optimization attractor. The agent optimizes for the public test signal — the only feedback available during its run. It stops when CI passes. It does not ask "what else should be true here?" because nobody told it there was more to check. A model that satisfies the visible assertion without satisfying the hidden contract is a false-green: public_pass=1, hidden_pass=0. The trust gap — public pass rate minus hidden pass rate — measures how often "looks fixed" is actually "is fixed."

This paper is the v2 update of the original Trust Gap report, now with pass@3 data — three independent fresh starts per scenario with no memory of prior attempts. The original v1 paper established the core finding across 5 models from 4 families: a consistent 20–62 point trust gap at pass@1, with three scenarios unconquered by any model. v2 narrows to a head-to-head pass@3 comparison of two models with identical architecture (3B-active MoE) but different training data and deployment — Laguna XS.2 (Poolside, OpenRouter free) and North Mini (Cohere, OpenCode free). Every scenario, prompt, and hidden test is identical. Every run was automated through the same harness.

1. The two models are indistinguishable on ci_forensics at pass@3 best-of-3. Both converge to 9/12 hidden pass with the same three unconquered scenarios.

2. Contract-visible prompts collapse the trust gap identically for both models on ci_forensics (to 5.6%) and eliminate it entirely for Laguna on maintenance_value (0%).

3. Retries do not converge. North Mini regressed from pass@1 to pass@2 — re-sampling from a distribution of wrong answers can produce worse outcomes.

4. Both models hit first-output timeout walls in different cells. The heavier contract-visible prompts cause the agent to fail to produce any output at all on product_workflows.

2. Method

2.1 Harness

Each scenario is a self-contained Python repository with a failing public test suite. The harness executes: (1) write scenario to fresh worktree, (2) run baseline tests to confirm red CI, (3) invoke the coding agent via OpenCode CLI with the rendered prompt, (4) run public tests to record CI state, (5) inject hidden acceptance tests, (6) run hidden tests, (7) store all artifacts. Each pass@k attempt is an independent fresh start — new worktree, no memory of prior attempts.[1]

3.6 The Deterministic False-Green: decimal_money_rounding

Laguna produced byte-identical wrong patches on all 3 attempts of decimal_money_rounding:

- total = 0.0
+ total = Decimal("0")
- total += float(line["unit_price"]) * int(line["quantity"])
+ total += Decimal(line["unit_price"]) * int(line["quantity"])
  return int(total * 100)  ← truncates toward zero, doesn't round half-up

The model independently produced the same wrong solution three times, each from a fresh worktree with no memory. This is not random variance. It is a stable attractor — the model converges to the same incomplete solution because the public test does not push it toward the hidden contract. The evaluator agent (DeepSeek V4 Pro) confirmed: int(Decimal("0.005") * 100) = int(Decimal("0.500")) = 0, truncating instead of rounding half-up to 1. Observed Evaluator verified, mean confidence 0.95, root cause = edge_case_gap

4. Discussion

4.1 The Trust Gap Is Structural, Not Model-Specific

Two independently trained 3B-active MoE coding agents from different companies produce identical false-green rates on identical scenarios. The pass@3 best-of-3 view collapses the apparent pass@1 gap to zero. On ci_forensics sparse, both solve 9 of 12 scenarios and fail the same 3. On ci_forensics contract-visible, both have exactly 2 false-greens across 36 runs. The trust gap is a function of the gap between the visible test and the hidden contract — not of model size, training data, or reasoning capability.

4.2 Contract-Visible Eliminates the Gap on Mechanical and Refactoring Tasks

PackSparse FG%CV FG%DeltaDelegable?
ci_forensics27.8%5.6%-22.2Yes

Explicit acceptance criteria collapse the trust gap from 27–28% to 0–6% on mechanical and refactoring tasks. Maintenance tasks are fully delegable with clear specs — Laguna scored a perfect 30/30 hidden pass. The intervention is the specification, not the model. When the agent knows what acceptance looks like, it can usually reach it.

4.3 The Spec Density Ceiling

Business logic resists even explicit spec exposure. Product_workflows sparse produced a 72.7% false-green rate with 7 of 11 scenarios completely unconquered at pass@3. The contract-visible lane for these tasks cannot even begin — every attempt hits the 600s first-output timeout. Either the prompt is too heavy (acceptance criteria for billing, discounts, inventory, SLAs, audit logging in a single prompt) or the tasks encode policy that cannot be fully specified without revealing the hidden tests. We cannot distinguish these explanations without further experiments.

4.4 Implications for Deployment

The evidence supports a three-zone deployment policy based on task type, not model choice:

Green Delegate with clear specs. Mechanical tasks (header contracts, parsing, rounding) and refactoring (migrations, renames, schemas). Contract-visible collapses trust gap to 0–6%. Both models perform identically.

Amber Gate on hidden acceptance. Adapters. Validators. Edge-case-heavy scenarios. Clear specs reduce but do not eliminate the gap. Run hidden acceptance tests before merge.

Red Do not delegate. Billing. Discounts. Inventory. SLAs. Audit logging. 72.7% false-green under sparse — 7 of 11 scenarios completely unconquered. Even the frontier model (DeepSeek V4 Pro at pass@1) only solved 3 of 11. The model is not ready. Human review is the trust gate.

5. Limitations

1. Incomplete matrix. North Mini has only ci_forensics (both lanes) complete. Maintenance sparse is partial (2/30 real runs). Maintenance cv and all product_workflows cells are not yet run. Cross-model claims beyond ci_forensics are limited to single-model evidence from Laguna. Not built 4 of 12 cells

2. 34 timeout-deleted runs. We deleted timeout rows (exit code 124) from two DBs to enable retry. These represent real operational signals — the model cannot begin some tasks under heavier prompt load. The deletion resets retry counters but the timeout wall itself is a finding. Observed 34 exit-code-124 rows deleted for retry

3. Incomplete evaluator coverage. Only 10 Laguna false-greens have been reviewed by the evaluator agent (all pass@1 ci_forensics sparse). The remaining 49 false-greens are unverified. Evaluator verdicts may not be reproducible without the same evaluator model. Not built evaluator review for pass@2, pass@3, maintenance, product, North Mini

4. System comparison, not model ranking. Laguna runs on OpenRouter free tier with rate limits. North Mini runs on OpenCode free tier with different rate limits. Speed asymmetry (17s vs 43s) is confounded by deployment surface. Timeout behavior differs by provider. We measure system behavior, not abstract model quality.

5. Both models share MoE architecture (3B active). The head-to-head finding that "model choice doesn't matter" may be specific to this architecture class. The v1 study (paper.html) found the same trust-gap pattern across 4 model families including dense models and a frontier model, suggesting structural rather than architectural cause. But the convergence-at-pass@3 claim is only validated for these two MoE models.

6. 33 scenarios. One harness. No external audit. The hidden tests and evaluator verdicts are authored by the same team. Tier 1 evidence — behavior microscope only. Tier 2 (comparative claims across models) needs evaluator agreement. Tier 3 (public benchmark) needs external audit.

6. Conclusion

Two independently trained coding agents from different companies produce identical false-green patterns at pass@3. The trust gap is structural — a function of the distance between the visible test and the hidden contract, not of the model. Contract-visible prompts eliminate the gap entirely on refactoring tasks (0% FG) and nearly eliminate it on mechanical tasks (5.6% FG). Business logic resists all interventions: 72.7% false-green under sparse, and the contract-visible lane cannot even begin.

Retries do not converge. The model re-samples from a distribution of wrong answers, and re-sampling can produce worse outcomes. North Mini regressed from pass@1 to pass@2. The marginal gain from retry on ci_forensics is +1 hidden pass for Laguna and +0 for North Mini. Three scenarios remain unconquered across 18 independent attempts.

The practical implication is clear: partition CI repair by task type, not by model. Mechanical work can be delegated with clear specs. Refactoring work can be fully delegated with clear specs — Laguna scored 100% hidden pass on maintenance_value. Business logic needs human review. The model is replaceable. The trust gate is not.
