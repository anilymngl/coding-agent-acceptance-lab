# Coding Agent Acceptance Lab

A reproducible harness for measuring a specific coding-agent failure mode:

> Did the agent actually satisfy the acceptance contract, or did it only make the visible CI tests pass?

The lab builds deterministic challenge repositories, runs a coding agent against visible tests, injects hidden acceptance tests only after the agent exits, and records whether a public-green patch is also acceptance-correct. A public-green / hidden-red run is a **false-green**. The main metric is the **trust gap**: public pass rate minus hidden pass rate.

This is a behavior microscope, not a public benchmark leaderboard. Treat model comparisons as evidence about this harness, these scenarios, these prompt lanes, and these runtime conditions.

## Main Finding

Visible CI pass is not the same as acceptance correctness. In the 391-retained-attempt Study B matrix, contract-visible prompting reduced false-greens substantially, but did not eliminate them.

Canonical recomputed Study B headline:

| Measure | Value |
|---|---:|
| Scenarios | 33 |
| Planned attempts | 396 |
| Retained attempts | 391 |
| Public-green attempts | 385 |
| Hidden passes | 284 |
| False-greens | 101 |
| Sparse false-green rate | 84 / 194 = 43.3% |
| Contract-visible false-green rate | 17 / 191 = 8.9% |
| North Mini product sparse false-green rate | 80.0% |
| North Mini product contract-visible false-green rate | 32.3% |

The result changes the deployment question from "did CI pass?" to "what acceptance gate makes this agent trustworthy for this task class?"

## Architecture

The core pipeline is:

1. write a deterministic scenario repository
2. run baseline tests and require red CI
3. run the coding agent with visible tests only
4. run public tests
5. capture the patch
6. inject hidden acceptance tests
7. run hidden tests
8. persist metrics, artifacts, and review metadata

Primary source paths:

| Path | Purpose |
|---|---|
| `ci_vibe_lab/scenarios.py` | scenario contracts, visible files, hidden tests, prompt lanes |
| `ci_vibe_lab/runner.py` | single-run pipeline |
| `ci_vibe_lab/matrix.py` | matrix expansion and orchestration |
| `ci_vibe_lab/report.py` | Markdown report generation |
| `ci_vibe_lab/evaluator.py` | evaluator workbench and review schema |
| `configs/matrix/` | reusable matrix definitions |
| `data/releases/v1/` | frozen public research data |
| `publishables/` | canonical publication source |

## Run Locally

Install dependencies:

```bash
uv venv
source .venv/bin/activate
uv sync --extra app
```

Run the no-model test suite:

```bash
uv run python -m unittest discover -s tests -v
```

List scenarios:

```bash
uv run ci-vibe-run list
uv run ci-vibe-run list --pack maintenance_value
```

Run a harness-only smoke without model tokens:

```bash
uv run ci-vibe-run run \
  --challenge dependency_api_change \
  --no-opencode \
  --db /tmp/ci-vibe-smoke.sqlite \
  --runs-dir /tmp/ci-vibe-smoke-runs
```

Inspect a stored run:

```bash
uv run ci-vibe-run inspect \
  --db /tmp/ci-vibe-smoke.sqlite \
  --latest \
  --full
```

## Project Origin

The project began as a North Mini Code pass@1 trust-gap investigation. That origin exposed the core false-green pattern and motivated the broader reusable harness, public release data, and two-stage Study B publication now represented here. Historical names and provenance identifiers may still mention `north-mini-test`.
