RESEARCH MVP / JUNE 2026

Context as Governed Infrastructure

Metric governance, authority boundaries, and the SRE approach to AI analytics.

An engineering discussion on taking a hackathon PoC toward a production-grade semantic layer on Snowflake.

THE PROBLEM

Agents hallucinate metrics by blending uncurated sources

When an AI agent answers a metric question, it finds many plausible sources. The failure is not retrieval. The failure is authority blending.

SourceAuthorityTypical agent behavior
Metric registry rowCertifiedBlended with all others
Semantic view definitionCertifiedBlended with all others
GitHub code chunkImplementationBlended with all others
Dashboard documentationReferenceBlended with all others
Live SQL outputObservedBlended with all others

The agent produces one confident answer without explaining which source is official. More context makes this worse, not better.

THE ARCHITECTURE

Three-lane evidence separation

Force the agent to separate sources by authority level. Never let them blend. Priority is enforced in code, not in the system prompt.

LanePurposeSource examplesOutput label
A Certified ContractWhat the metric officially meansSemantic views, ratified metric contractscertified
B Source EvidenceWhere the metric appears in code or docsGitHub READMEs, dbt model descriptionsreference_evidence
C Warehouse ObservationWhat the data currently showsLive queries, row sampleswarehouse_observed

Certified context says what the metric means. Source evidence shows where related text appears. Observed data shows current values. These are not the same authority level, and the agent must label them separately.

SOURCE OF TRUTH

Semantic views as the ultimate governed artifact

Semantic views are not just query targets for Cortex Analyst. They are the authoritative, versioned, owned expression of business logic. Every other source — documentation, code, dashboards — is a secondary perspective on the same contract.

PropertyRequirementWhy
OwnerAssigned, not nullAccountability for accuracy
Revision dateUpdated on every changeStaleness detection
Verified queriesAt least one golden questionProves the SV expresses intent correctly
Custom instructionsDisambiguation for similar labelsPrevents Cortex Analyst from guessing between near-identical columns
Grain and additivityExplicitly documentedPrevents incorrect aggregation across dimensions
Overlapping logicFlagged and cross-referencedDetects partial overlap between SVs that could produce conflicting numbers

Centralizing on semantic views shifts the pipeline's core value. Ingestion and cross-SV logic mapping become more valuable than raw data access. The SV is the contract; everything else is commentary.

TEMPORAL DRIFT

Code and contract diverge silently over time

The certified contract says Revenue = X. The dbt model pushed yesterday says Revenue = X + Y. The agent only reads the contract and gives the wrong answer with high confidence.

metric_contract.yaml — Lane A (ratified 2026-05-01)

definition: "Sum of closed_won opportunities where status = ACTIVE"

fct_revenue.sql — production dbt (pushed 2026-06-10)

- WHERE status = 'ACTIVE'
+ WHERE status IN ('ACTIVE', 'PENDING_RENEWAL')

Without AST-based ingestion that parses actual SQL to extract CTEs, WHERE clauses, and GROUP BY grains, the system cannot detect that the contract and the code have diverged. Documented intent and runtime reality become two different truths.

RETRIEVAL FAILURE

Vector search returns semantically similar, mathematically wrong chunks

Vector embeddings map semantic proximity, not logical equivalence. A question about "Gross Margin excluding hardware" retrieves the chunk for "including hardware" because the embedding distance is tiny.

QuestionRetrieved chunkCorrect?
Gross Margin excl. hardwareGross Margin incl. hardwareNo
Active Users (30d window)Active Users (7d window)No
Bookings by region EMEABookings by region globalNo
Churn rate (enterprise)Churn rate (SMB)No

Deterministic routing must filter before the LLM sees the context. Prompt-level instructions are fragile; architectural constraints are robust. A Router + Skill Manifest pattern replaces speculative retrieval with bounded injection.

TRANSPORT LAYER

MCP as a dumb pipe; governance lives in Snowflake

MCP is the serving boundary. It does not decide whether a metric is certified. The governance data model, stored procedures, and answer contract decide. Transport is decoupled from truth.

ConcernLives in
Tool exposure to external agentsMCP server
Certification status and routing prioritySnowflake registry + procedures
Evidence labels and authority levelAnswer contract (JSON payload)
Auth, RBAC, and scoped accessSnowflake roles + OAuth/SAML

Whatever the transport, agents must receive the same evidence labels, routing rules, and caveats. The governance data model is the stable interface; MCP is just one delivery mechanism.

External agent auth against Snowflake-hosted MCP is an open integration question. OAuth2 PKCE flows, decoupled identity layers, and scoped read-only API keys are likely required. Dedicated spike needed before committing to external-facing architecture.

ROUTING MODEL

Progressive skill loading over naive vector search

Replace open-ended retrieval with a Router + Skill Manifest pattern. The agent downloads a bounded skill file, not 15 chunks of blended Markdown from five repos.

StageNaive RAGSkill Registry
Intent detectionImplicit in the questionExplicit router classifies domain
Context sourceTop-k vector chunksSkill manifest file from registry
Authority scopeMixed in system promptBounded per skill, no cross-domain leakage
Failure modeConfident blendingWrong skill selected (detectable)

skill_finance_revenue.yaml

system_prompt: "...use Lane A only when certified contract exists..."
allowed_contracts: [rev_gaap, rev_bookings]
semantic_view: sem_finance_revenue
boundaries: "no supply chain data, no HR data"

CONTEXT ISOLATION

Context sandboxes prevent experimental logic from leaking

When multiple metric variants exist, the agent must not blend them. Isolate each variant into its own context sandbox with explicit justification for differences.

MetricVariantJustificationStatus
base_active_users—Umbrella definitioncertified
base_active_users7d_windowProduct team engagement metricjustified
base_active_users30d_windowFinance retention KPIjustified
base_active_usersmobile_onlyExperimental, not yet ratifieddeprecated

Secondary sources never become accidental sources of truth. Precedence is explicit and enforced: certified > justified > deprecated. Unratified variants cannot silently override the umbrella definition.

THE ACTUAL PRODUCT

The Governance Workbench UI is the real deliverable

The AI agent is the read-only consumer. The actual value is the surface where humans ratify, review, and resolve metric collisions. If the governance workflow is a stored procedure, adoption dies in week two.

SurfaceFunction
Collision InboxGroups conflicts between teams' metric definitions, shows exact SQL and contract diffs
Ratification FlowOne-click approve umbrella metric, demote conflicting variants, force justification
Eval TriggerFires the CI/CD eval battery on every ratification to prove agent routing preserved
Agent View SimulatorSplit-screen trace showing steward exactly what the agent saw and which sources it used
Health DashboardOwner, revision date, and staleness across all governed artifacts

Without the workbench, stewards ignore governance and the "Certified" label becomes a graveyard of outdated metadata.

TRUST SURFACE

Agent View Simulator: stewards see what the agent sees

A split-screen where the steward types a question and inspects the exact routing chain. Builds trust faster than any chatbot demo by showing the agent did not blend sources.

Query: "What was Q2 gross margin in EMEA?"

Router → finance_gross_margin skill
Lane A → metric_gross_margin_gaap (certified, owner: jane.doe)
Lane C → sem_finance_margin view (warehouse_observed)
Lane B → skipped (Lane A certified contract present)
SQL → SELECT ... FROM sem_finance_margin WHERE region = 'EMEA'

The steward verifies authority separation without reading agent logs. Every routing decision is visible and auditable.

VERIFICATION

CI/CD Eval Harness scores routing, not just answers

Manual Slack demos do not catch regressions. An automated battery runs on every context file or semantic view change and blocks the PR if the agent breaks.

AssertionPass criteria
Contract tool calledBefore SQL generation
Authority label presentJSON contains authority_level
No blended sourcesOnly one evidence lane in response payload
Ambiguity surfacedCaveat emitted when metric has multiple variants; no silent hallucination
Missing metadata reportedExplicit null in owner/revision fields, not invented

Every regression points to a fixable owner: registry, search index, ingestion pipeline, Cortex Analyst, SQL generation, routing logic, or synthesis. No mystery failures.

THE ORGANIZATIONAL SHIFT

From documentation project to semantic SRE

Context staleness is a production reliability issue, not a documentation gap. Semantic views, metric contracts, and verified queries are first-class production assets with SLAs and owners.

Traditional modelContext engineering model
Data engineers own tablesData engineers own semantic views
Docs are extra work, optionalContext files are load-bearing infrastructure
Broken dashboard is a Jira ticketBroken semantic view is a P1 incident
Stale docs are tolerated indefinitelyStale context triggers an SRE alert

Breaking a semantic view is as serious as breaking a production API. The blast radius is every AI agent that consumes it.
