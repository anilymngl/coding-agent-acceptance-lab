Hidden
02

THE QUESTION NOBODY ASKS

When an AI makes a test pass — did it actually fix the problem? Or just the one thing the test checked?

Here's the thing. The AI sees a failing test. It edits code. The test passes. Green light. But — is the green light real, or did the AI just make it green by doing the minimum? The hidden test answers that. It checks the same problem from a different angle. Edge cases. Other places the same mistake might be hiding. Things a good engineer would check naturally — but the AI only checked what was in front of it.

A real example: timezone migration

The project has a function created_at() that uses an old, deprecated way to get the current time. The visible test checks: does created_at() return a proper timezone-aware datetime?

What the AI sees (the failing test):

def test_timezone_aware():

value = created_at()

assert value.tzinfo is not None ← FAILS

The AI fixes created_at(). The test passes. CI is green.

→

What the AI missed — the hidden test reveals it

After the AI exits, the hidden test is added. It checks something the visible test didn't: does the updated_at_iso() function in the same file also have the same problem? And are there any remaining calls to the deprecated method anywhere?

The hidden test (AI never saw this):

def test_updated_at_also_fixed():

parsed = datetime.fromisoformat(updated_at_iso())

assert parsed.tzinfo is not None ← STILL FAILS

def test_no_deprecated_calls():

code = open("app/timestamps.py").read()

assert "utcnow()" not in code ← STILL FAILS

The AI fixed one function. It didn't check the other function in the same file. A human would have. The hidden test catches this gap. Visible green. Hidden red. Fake fix.

So the hidden test is not a trick. It's not something extra the AI "shouldn't need to know."

It's the same requirement — "make timezone-aware timestamps" — applied more thoroughly. A good engineer would check the whole file. A good engineer would search for other uses of the deprecated method. The hidden test just checks whether the AI did what a good engineer would do — or stopped at the first green light.

Scenario
03

THE BUILDING BLOCK — ONE SCENARIO

37 hand-crafted test cases. Each one has: broken code, a failing visible test, a hidden acceptance test, and the rules for what "done" means.

What every scenario contains

Broken source code

The file that needs fixing. Written into a fresh project folder.

Failing visible test

The test the AI sees failing. This is what the AI tries to make pass.

Hidden acceptance test

Tests the same requirement more thoroughly. Injected only after the AI finishes. Edge cases, other locations, compatibility checks.

Expected behavior

What a correct fix should look like. Shared with the AI when you give clear instructions. Hidden when you don't.

Known failure modes

Common wrong approaches the AI might take. Also shared when you give clear instructions.

Three ways to present the same scenario

Vague ticket

"CI is red. Here's the error. Fix it." The AI has to figure out what "done" means by itself.

Clear instructions

Same as above, plus the expected behavior and common mistakes. Tests whether the AI can follow the spec.

Reviewer context

The trap, difficulty level, tags. Only the independent reviewer sees this version — never the AI being tested.

The gap between vague and clear tells you: is the AI incapable, or just uninformed?

The hidden test is mechanically enforced

tests/test_hidden_acceptance.py

This file does NOT exist while the AI is working.

It gets written to disk only after the AI finishes.

Then all tests (visible + hidden) run together.

Both pass → real fix.

Visible passes, hidden fails → fake fix.

The AI cannot see this file. Cannot read it. Cannot optimize for it. It's not a hint the AI "should have checked." It's a measurement of what the AI did — applied after the fact.

Runner
04

WHAT HAPPENS WHEN A SINGLE TASK RUNS

Five steps. Everything recorded. Nothing thrown away.

Step 1: Prepare

Make a fresh copy of the project. Write the broken source file and the failing visible test. Run the test. It must fail. If it passes, the scenario is broken — stop.

Saved: baseline.txt

→

Step 2: Let the AI try

Give the AI the project folder and the prompt. It sees the failing test output. It edits files. It runs commands. It can read, write, search — whatever it wants. The hidden test does not exist yet.

Saved: prompt.txt, opencode_stdout.jsonl

→

Step 3: Check visible tests

Run the original test again on whatever the AI left behind. Did it pass? Record the answer. No judgment yet — just: did the AI make the green light?

Saved: public.txt, patch.diff

Step 4: Inject hidden tests

Now — and only now — write the hidden acceptance test into the project. Run ALL tests (visible + hidden). The AI never knew this second check existed. It couldn't have prepared for it.

Saved: hidden.txt

→

Step 5: Store everything

One row in the database. Files on disk. The prompt, the AI's entire work log, both test results, the code changes. Absolute paths. Traceable. If someone doubts a number, they can open the exact files that produced it.

Saved: SQLite row + artifact folder

What you get per run

prompt.txt

baseline.txt

public.txt

hidden.txt

Labels
05

AFTER EACH RUN — THE SYSTEM DECIDES WHAT HAPPENED

Every result gets exactly one label. The logic is mechanical — the AI doesn't decide its own score.

The AI produced usable output → these measure quality

Real fix ✓

Visible test passed. Hidden test passed. It actually works. Ship it.

Fake fix ✗

Visible test passed. Hidden test failed. The AI made it look done. It isn't. This is what the whole harness exists to measure.

Couldn't pass ⬜

The AI couldn't even make the visible test green. Nothing to merge. Nothing to review.

The infrastructure failed → these measure reliability, not quality

Timed out (working) ⏱

The AI was actively working — produced output, made edits — but ran out of time.

Timed out (stalled) ⏱

Zero output before timeout. Connection issue, rate limit, or model overload. Not a wrong answer — no answer at all.

Setup errors ⚠

Wrong model name. Network failure. Auth error. Test runner broke. Nothing to do with the AI's ability.

Why separating these matters

Most evaluations mix "the AI produced a wrong answer" with "the AI couldn't respond in time" into one number. That's how you get garbage. Only the left column feeds the fake-fix rate. The right column tells you about deployability.

Review
06

I DON'T ASK A SECOND AI "IS THIS GOOD?" — THAT'S JUST VIBES.

When a fix is fake, a separate AI reviews it. But it has to prove its judgment.

What it receives

A folder with everything: the original project, the AI's changes, the hidden test, the AI's full work log. Any claim it makes must quote something in this folder.

EVALUATION_PACKET.md

Original code

The AI's changes

Hidden acceptance test

Full work log (every file read, every edit)

The prompt text

Both test results

→

How it proves the AI was wrong

It gets a blank workspace. It tries to write its own fix that passes the hidden test. If it can do it, the original AI failed. If even the reviewer can't do it, maybe the test itself is broken. Proof by demonstration, not opinion.

workbench/

model_repo/ (original)

shadow_repo/ (blank — reviewer writes here)

→

What it produces

A structured report. Machine-verified — every quote must exist verbatim in the evidence. If it cites something that isn't there, the system rejects the report.

Real example: The AI fixed math in the raw data layer. The correct place was the display layer. The reviewer proved this by writing the fix in the right place — hidden tests passed. The AI's own logs showed it never checked where the result was displayed. Four pieces of evidence cited. 95% confidence.

FROM DATABASE ROWS TO READABLE OUTPUT

Three types of reports. One integrity check. One live dashboard. All from the same database.

Comparison (leaderboard)

The main output. Per-model per-pack per-instruction-style tables. Fake-fix breakdowns with reviewer verdicts. Operational reliability scores. What can and cannot be defended from the evidence.

Integrity check

SHA256 hash of every artifact file. Verifies every path stored in the database exists on disk. Checks reviewer coverage — what % of fake fixes got reviewed. Flags anything suspicious.

Done
10

ARCHITECTURE SUMMARY

Everything is traceable.
Everything is verifiable.
Everything fits in one weekend.

What's different

Hidden tests are mechanically enforced — not a convention, a code path. The file doesn't exist while the AI works.

Reviewer must prove its claim — write a correct fix, not just opine. Every quote verified verbatim.

Results classified automatically — real fix, fake fix, couldn't pass, or infrastructure failure. No AI judging its own score.

Matrix isolates variables — model, prompt, work type. One-cell anecdotes can't distinguish causes.

Claim system has explicit gates — you publish at the tier you've earned evidence for.

What's simple

Python. SQLite. JSON. Bash.

No microservices. No queues. No cloud.

One command to run. One to watch. One to report.

All data is files on disk. All analysis is SQL queries. All reports are markdown.

Built in one weekend. 37 scenarios. 132 runs. 6h50m.

What's next

Auto-retry stalled runs — delete timeouts, resume, keep real results.

Multiple attempts — measure consistency, not one lucky try.

Independent audit — someone else reviews the hidden tests.

Two reviewers — compare judgments, measure agreement.

One-click study — run, retry, review, report in one command.
