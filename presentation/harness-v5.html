Problem
02

EVERY POPULAR BENCHMARK MEASURES THE WRONG THING

They count how often tests pass.
That's exactly backwards.

If you reward an AI for making tests green, it learns to delete checks, hide errors, and return fake values. The test passes. The problem doesn't go away. Every major benchmark treats "looks done" as "is done." Nobody measures how often that's wrong.

01

Standard tests measure writing new code

"Write a function that does X." No existing project to break. No wrong-but-plausible fix. No test you can cheat on.

02

Real work is "this test is failing — fix it"

The task is editing existing code in an existing project. The danger isn't producing nothing. The danger is producing something that looks correct and isn't.

03

AI stops when the test turns green

The failing test is all the AI can see. It doesn't know there are hidden requirements. It stops as soon as the light changes. Reward it for that, and you train it to stop early — every time.

If you use an AI assistant without hidden quality checks,
you are trusting a signal that is wrong 30-40% of the time.

How
03

TESTS THE AI SEES. TESTS IT NEVER SEES.

Two sets of tests. One question:
did the fix actually work?

Visible tests = what a real project shows the AI during its work. Hidden tests = extra checks added only after the AI finishes, checking things the visible tests missed. If both pass → real fix. If visible passes but hidden fails → the AI made it look fixed without actually fixing it.

Actually fixed ✅

The fix is real.

Both sets of tests pass. The AI understood the problem. The change is safe to use.

Looked fixed but wasn't ❌

The test passed. The bug stayed.

The AI made the visible test green — by deleting a check, catching an error silently, or returning a fake value. The test lied to you.

Didn't even pass ⬜

Couldn't make the test green.

The AI couldn't produce a fix that even passed the visible test. Nothing to review.

The number that matters: how many passing fixes were actually wrong.

If 10 changes pass the tests but 3 are fake, that's a 30% deception rate. Not "accuracy." How often the green light lies. This separates "can the AI code" from "can the AI code correctly" — two things every other test blends into one number.

Every test run gets one of eight labels:

✓ real fix ✗ fake fix ⬜ couldn't pass
⏱ timed out ⏱ no response
⚠ setup error ⚠ tool error
— not run yet

"Couldn't produce an answer" and "produced a wrong answer" are different problems. Don't mix them.

Verify
04

I DON'T ASK "HEY AI, IS THIS PATCH GOOD?" — THAT'S JUST VIBES.

A second AI checks the work.
But it has to prove what it claims.

When a fix looks correct but isn't, a separate AI receives the entire project: the original code, the attempted fix, the hidden tests, and a blank workspace. It tries to write a correct version and runs the hidden tests against it. If it can fix it, the original was wrong. If it also can't, maybe the test itself is broken. Every claim it makes must be backed by an exact quote from the evidence — automatic checks enforce this. AIs make things up. This makes that impossible.

Evidence File

All the code, all the logs, both test results, everything the AI did. The reviewer can only reference what's in here.

Test Workspace

The reviewer writes its own fix in a fresh workspace and runs the hidden tests. If it passes, the original AI failed. Proof, not opinion.

Judgment

Structured report: what went wrong, how bad it is (1-5), how confident the reviewer is (0-1), exact quotes from the evidence. No hallucination possible.

Real example — the AI fixed the math in the wrong place (95% confidence)

The AI put the correction in compute(), but the correct place was dashboard_total(). Right math, wrong location. The evidence proves it — the AI's own notes show it never checked where the result was displayed.

Vague ticket ("the test is red, fix it") vs Clear spec (here's exactly what needs to work).

If the spec helps, the AI is capable but uninformed. If it doesn't help, the task is genuinely hard.

Guess the rules vs follow the rules.

Some work is mechanical. Some needs understanding of money math, time zones, and user permissions.

Finding
07

THE BIG DISCOVERY

Clear instructions help both AIs.
But in completely different ways.

North Mini — already passes most tests

Instructions turn fake fixes into real ones.

This AI almost always makes the test green. The problem isn't getting it to pass — it's that some of those passes are fake. Clear instructions convert existing passes from fake to real. Simple fixes: 4 fake → 1 fake. Refactoring: 2 fake → 0 fake. Zero fake fixes on refactoring with clear specs.

Its weakness: fixes the specific test, not the underlying rule. Tell it the rule explicitly, and it delivers. Fake rate drops from 20-33% to 0-8%.

Vague: passed test / hidden tests failed
Spec: passed test / hidden tests passed

Gemma 12B — often can't pass tests at all

Instructions help it produce working code.

This AI frequently cannot even make the test green on its own — 7 of 12 bug-fix tasks failed under vague instructions. Clear specs act as scaffolding — they help it build a working fix where it couldn't before. Bug fixes: test passes went from 5 to 7, real fixes doubled from 3 to 6. Business logic: 7 real fixes out of 11 — the single best result on the hardest work type.

Its weakness: navigation — it struggles to find the right code in a foreign tool. The spec helps it know what to look for. Fake rate drops from 16-36% to 8-10%.

Vague: couldn't pass tests
Spec: passed test / hidden tests passed

Same improvement. Different reason. That's exactly why you test both AIs in both conditions — one result tells you instructions help. The comparison tells you why — and for whom.

Action
10

HERE'S WHAT THE DATA SAYS YOU SHOULD ACTUALLY DO

Three buckets. The question is: how much hidden complexity does the task have?

Safe

Trust the green light

Simple, mechanical changes. The visible test nearly covers everything that matters. Both AIs reliable here — especially with clear instructions. Zero fake fixes for the faster AI with specs on refactoring.

✓ regenerate API docs

✓ update test fixtures

✓ rename deprecated methods

✓ fix broken imports

✓ sync documentation

Caution

Hidden check required

The visible test is incomplete. The AI can make it pass, but you need a hidden check to catch gaps. Clear instructions convert fake fixes to real ones in this bucket.

⚠ field renames (old data? new names?)

⚠ data splitting (empty input? bad sizes?)

⚠ boolean parsing (blanks? whitespace?)

⚠ CSV headers (all columns? ordering?)

Danger

Human review required

Complex business rules. AIs produce surface-level fixes that miss core requirements. Even clear instructions don't fully close the gap.

✗ billing calculations

✗ discount stacking order

✗ inventory state changes

✗ SLA business hours

✗ rounding money amounts

The workflow: write clear spec → let AI attempt → hidden check → if it fails, get a human or a stronger AI → replace the AI, keep the check. The process outlasts any model.

Claims
11

EARLY EVIDENCE — NOT A FINAL VERDICT

What I Stand Behind

  • ✓ The two-test system (visible + hidden) is mechanically enforced. Every claim links to a specific test run in the database.
  • ✓ The fake-fix pattern appears across four different AI models. Same tasks fail in the same way across different engines.
  • ✓ Clear instructions measurably reduce fake fixes. The effect is consistent, directional, and visible.
  • ✓ Refactoring tasks under clear specs are a safe starting point. Zero fake fixes for the faster AI. High success for both.
  • ✓ The three hardest tasks reveal a missing-information problem, not an inability problem. Instructions cracked two of them.

What I Cannot Claim (Yet)

  • ✗ One AI is better than the other. The faster AI runs on its native tool. The local AI runs through unfamiliar software. This compares systems, not raw ability.
  • ✗ Single attempts are reliable. The 30-point swing proves they're not. You need multiple attempts per task to characterize typical behavior.
  • ✗ Business logic tasks are fully measured. The faster AI's results are incomplete due to free-tier limits. The local AI's data is clean; the other's isn't.
  • ✗ This has been externally reviewed. 37 tasks, one person, one reviewer AI. Early-stage evidence only. Independent review would be needed to publish as a benchmark.

This is first-stage, honest evidence. The method is what's impressive here — not the specific numbers. Built in one weekend. This is what evaluation should look like.
