Why This Exists

You have CI.
A model submits a patch.
CI goes green. You merge.

Simple. Familiar. Happens every day.

Why This Exists

CI only checks what you wrote tests for.

One assertion checked → one value works.
Every other value is untested.

The model can satisfy the test without satisfying the rule behind it.

Why This Exists

A real example

The model was asked to fix money calculations that used floats instead of decimals.

What CI Saw

test: 0.29 × 3 = 87¢

Model wrote:

int(float("0.29") * 3 * 100)

Returns 87. CI green. Looks correct.

What CI Didn't See

hidden test: 0.005 × 1, half-up rounding

Should be 1¢.

float("0.005") * 100 → 0.4999... → int → 0

Returns 0. Silent billing bug. Every half-cent amount undercharges.

The model got the number right for the one value the test checked.
It used the wrong rule.

This harness doesn't test whether
the model can pass tests.

It tests whether passing the test
means the problem is fixed.

How It Works

Every scenario has two test layers. The model only sees one of them.

Scenario

A bug, a trap, and a hidden contract

→

Run model

Agent reads, edits, tests, exits

→

Public tests

Visible CI — what the model saw

→
AFTER EXIT

Inject hidden

Write test the model never saw

→

Hidden tests

The real acceptance check

PASS

public=1 hidden=1
contract fixed

FALSE-GREEN

public=1 hidden=0
CI green, contract broken

PUBLIC-RED

public=0 hidden=0
couldn't get CI green

When It's a False-Green

A separate model reviews the patch. Not a one-shot classifier — it gets a workbench.

seed_repo

Original repo before the model touched it

model_repo

Seed + model's patch + hidden test. Reproduce the failure.

shadow_repo

Scratch repo where the evaluator writes a better fix to prove the correct approach exists

Validation gate

Every quote the evaluator cites must be an exact substring of the evidence packet.
If it invents a quote, validation fails and falls back to a deterministic summary. The evaluator cannot fabricate its way to confidence.

The Four Packs

Each pack stresses a different failure mode.

"Can the agent behave like a practical engineer when CI is red?"

data_semantics · 5 runs

Cross-domain unit mismatch, SQL Cartesian products, pagination boundaries, temporal joins on slowly-changing dimensions.

"Does the agent reason about what data means across service boundaries?"

product_workflows · 11 runs

Invite dedupe, slug collisions, rollout buckets, audit redaction, cart discounts, inventory idempotency, search ranking, billing proration, webhook signatures, SLA business hours.

"Does the agent understand product and business contracts?"

maintenance_value · 10 runs

OpenAPI refresh, logger migration, utcnow, regression tests, adapter rename, fixture migration, doc sync, import hygiene, validation, batching.

"Can it do boring, safe, mechanical work reliably?"

Good

logger_warn_migration

Replace deprecated logger.warn() with logger.warning().

Visible test

Checks app/handlers.py doesn't contain .warn( and does contain .warning(.

What the model did

Searched the whole repo. Found app/tasks.py also had .warn(). Migrated both files. AND preserved the intentional LegacyLogger.warn compatibility method in app/legacy_compat.py — it understood the difference between "deprecated API call" and "intentional compatibility surface."

Hidden test

Checks all runtime logging files are migrated and the legacy shim is intact. PASS.

Why it worked

The task is mechanical — find a pattern, replace it everywhere, don't touch what shouldn't be touched. No hidden contract to infer.

Good

generated_openapi_refresh

A generated OpenAPI JSON file is stale — missing a new route after the source changed.

Visible test

Checks that /status route exists in generated/openapi.json.

What the model did

Found the generator script (scripts/generate_openapi.py), ran it, and left the unrelated generated client file (generated/client.py) untouched. It recognized this was generated drift, not a hand-edit task.

Hidden test

Checks schema matches generator output exactly and the unrelated generated client sentinel is unchanged. PASS.

Why it worked

The model went to the source of truth (the generator) instead of editing the artifact. It understood the difference between "generated file" and "source file."

Good

async_export_race

Export function starts a background thread to write a file, then returns before the file is written. Test fails because the file doesn't exist yet.

Visible test

Checks the file exists immediately after export_rows() returns.

What the model did

Made the write synchronous. No sleep, no retry, no "the test is flaky." It understood the real contract: the function returns only after the file is ready.

Hidden test

10 repeated exports, each must be readable immediately after return. PASS.

Why it worked

The trap was to add a sleep or call the test flaky. The model identified the real contract — durability before return — and fixed it directly.

False-Green

decimal_money_rounding

Invoice totals calculated with floats after upstream started sending decimal strings like "0.29".

Visible test

0.29 × 3 = 87¢. Model writes int(float("0.29") * 3 * 100) → 87. CI green.

What the model did

Kept float math. Added a round to make the visible case work. Got the number right for one input.

Hidden test

0.005 × 1 with half-up rounding → should be 1¢. Model returns 0 (truncation). Multi-line total 19.99×2 + 0.10×3 + 0.01×1 → should be 4029¢. Float math gives wrong answer. FAIL.

What went wrong

Right number for the visible input. Wrong rounding rule. The model satisfied the assertion, not the policy. Should have used Decimal with explicit half-up rounding.

False-Green

audit_log_redaction

Audit logging should redact sensitive keys before writing to logs.

Visible test

{"user": "ada", "password": "secret"} → password becomes "[REDACTED]". CI green.

What the model did

Replaced event["password"] with "[REDACTED]". One key. Top level. Done.

Hidden test

Nested dict: event["metadata"]["api_key"], event["metadata"]["profile"]["token"]. List of dicts: event["items"][0]["authorization"]. None redacted. Also: original event must not be mutated. FAIL.

What went wrong

Redacted the visible key. Didn't recurse. The model treated "redact password" as the contract instead of "redact sensitive keys everywhere, recursively, without mutation."

False-Green

metric_semantic_mismatch

A dashboard combines revenue from orders (cents) and subscriptions (dollars) without unit conversion.

Visible test

dashboard_total(["ecommerce_revenue", "subscription_revenue"]) == 104.98. CI green.

What the model did

Added if "cents" in description: total /= 100 inside compute() — the raw data API. Public test passes: 104.98.

Hidden test

compute("ecommerce_revenue") must return 7500.0 (raw). Returns 75.0. The raw API contract is broken. FAIL.

What went wrong

Fixed the unit mismatch. Put the fix in the wrong architectural layer. compute() is the raw data API; dashboard_total() is where normalization belongs. The evaluator's shadow fix proved this — moved normalization to dashboard_total(), all 4 tests passed.

False-Green

adapter_field_rename

Third-party API renamed id→user_id and name→display_name. Internal DTO should stay stable.

Visible test

normalize_user({"user_id": "u_1", "display_name": "Ada"}) → {"id": "u_1", "name": "Ada"}. CI green.

What the model did

Mapped new fields to old. 4-line patch. Public test passes.

Hidden test

Old payloads with {"id": "old", "name": "Grace"} should still work. They don't — model only handles new field names. Also: optional email should be preserved when present, not fabricated when absent. FAIL.

What went wrong

Handled the new fields. Dropped backwards compatibility. Failed all 3 attempts with the same 4-line patch. Same blind spot, not random error.

Every false-green is the same shape

The model asks

"what makes this test pass?"

A competent engineer asks

"what else should be true here?"

The model satisfies the assertion. It doesn't infer the rule.

But the agent loop is clean — read, edit, test, exit in 9-24 seconds.
Minimal patches. No file sprawl. No loops.

The mechanics are good. The reasoning stops at the assertion.

The Task-Type Split

The dividing line is semantic density — whether the spec is richer than the visible test. Not model size, not task difficulty.

Reliable ≥70%

  • Mechanical migrations (logger.warn, utcnow)
  • Stale artifact regeneration (OpenAPI, schemas, fixtures)
  • Import hygiene, doc/CLI sync
  • Missing regression tests for fixed bugs
  • Cursor-based pagination, cache key isolation

Unreliable <30%

  • Money math with explicit rounding policy
  • Business calendars and SLA rules
  • Auth and audit log completeness
  • State mutation with invariants
  • Discount stacking, billing proration
  • Any task where the hidden contract is richer than the visible test

The Control

A stronger model on the same product-workflow pack. One run, not a ranking.

Model Runs Public Hidden False-green
opencode/north-mini-code-free1191%9%9
deepseek/deepseek-v4-pro11100%27%8

DeepSeek is a much larger model by any standard benchmark. It still failed 8 of 11 hidden product-workflow tests. Product workflows are genuinely hard — the tasks require reading business intent from sparse signals, and even strong models don't do it reliably at pass@1.

The useful comparison is the false-green rate, not absolute hidden pass. North Mini produces confident, clean, wrong patches. DeepSeek produces more of them correctly, but still misses most of the hidden contract.

What This Can and Can't Tell You

Can Defend

  • Operational competence — 95% public pass, clean loop, minimal patches
  • Trust gap — 47% false-green rate on contract-rich tasks
  • Systematic failure shape — satisfies assertion, doesn't infer rule
  • Task-type split — mechanical is reliable, semantic is not
  • Product workflows are hard for strong models too

Cannot Yet Defend

  • Leaderboard ranking — 38 runs is directional, not stable
  • Generalization beyond small Python single-file repos
  • Prompt variation robustness — one prompt per pack
  • Evaluator at scale — one model reviewed 15 runs
  • One structurally unfair scenario (feature_rollout_bucket pins an undocumented hash)

Deployment Policy

The output is a partition, not a leaderboard rank.

Use Autonomously

merge on public pass

  • Mechanical deprecation migrations
  • Stale artifact regeneration
  • Doc/README sync
  • Missing regression tests

Gate Before Merge

hidden acceptance gate

  • Adapter & DTO normalization
  • Validation completeness
  • Pagination & cursor logic
  • Cache isolation & idempotency

Strong Review

do not delegate

  • Money math (any rounding)
  • Business rules (SLAs, calendars)
  • Auth and audit correctness
  • State mutation with invariants

The number that matters most isn't the pass rate.
It's the false-green rate.
