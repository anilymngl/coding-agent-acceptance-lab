North Mini Test

A coding-agent evaluation harness. Does the model fix the problem, or does it just make CI green?

The Question

Standard benchmarks score pass/fail. This harness scores how often you can trust a pass.

Every scenario has two test layers. Public tests are visible to the model during its run — like normal CI. Hidden tests are injected after the agent exits and never seen. A model that satisfies the public tests without satisfying the hidden ones is a false-green: it made CI green without fixing the contract.

38Runs

47%False-green rate

17 of 36 public-green patches were still contract-broken. That's the number that matters — not the pass rate, but how often a pass you'd merge is actually wrong.

What this is not
Not a leaderboard benchmark. 38 runs is a directional study, not a statistically stable sample. All scenarios are small Python repos with single-file fixes. The findings describe one model in one harness — treat them as a hypothesis generator, not a generalization.

The Evaluator Agent

A separate model reviews the false-green. Not a one-shot classifier — it gets a workbench.

Why an agent, not a prompt
The evaluator reproduces the failure, forms a contract hypothesis, and optionally writes a better fix in a scratch repo to confirm its judgment. A verdict backed by a shadow fix that passes every hidden test is a different kind of evidence than a verdict backed by vibes.

The workbench

RepoContentsPurpose
seed_visible_repo/Original challenge repoReference — what the model started with
model_repo/Seed + model's patch + hidden testReproduce the false-green failure
shadow_repo/Seed + hidden test (scratch)Evaluator writes a better fix here to prove the correct approach exists

Validation

The evaluator's output is validated, not trusted. Three layers:

  • Schema: Pydantic with extra="forbid", bounded enums, scores 1-5, confidence 0-1
  • Verdict consistency: a public-green/hidden-red run must use verdict public_green_hidden_red
  • Evidence: ≥2 objects, must cite hidden_test_output and patch or challenge_contract. Every quote must be an exact contiguous substring of the evaluation packet.
Example verdict — metric_semantic_mismatch

"Model applied cents-to-dollars normalization inside compute() (the low-level raw API) instead of dashboard_total() (the user-facing layer)." Confidence: 0.95. Patch quality: 2/5. Shadow fix passes all 4 hidden tests.

If validation fails, the harness falls back to a deterministic summary and preserves the raw agent output. A bad evaluator response cannot hide the run.

95% public pass looks excellent. 50% hidden pass is the real number. The 45-point trust gap is the deployment-changing signal.

The maintenance pack used best-of-3 (three attempts per task, smallest passing patch selected). First-attempt and best-of-3 results were identical — every scenario either passed hidden on attempt 1 or never passed.

The Failure Shape

The model asks "what makes this test pass?" instead of "what else should be true here?"

Across 17 false-greens, the same pattern every time: the model satisfies the specific failing assertion rather than inferring the rule behind it.

decimal_money_rounding
Got the number right for the visible input (0.29 × 3 = 87¢). Used the wrong rounding mode — truncated instead of half-up. The visible test checks one value; the hidden test checks the rounding policy.
audit_log_redaction
Redacted the visible top-level password field. Didn't recurse into nested dicts, lists, or other sensitive keys (api_key, token, authorization). The visible test checks one key; the hidden test checks recursive redaction without mutation.
metric_semantic_mismatch
Fixed the unit mismatch by dividing cents by 100. Put the conversion inside compute() — the raw data API — instead of dashboard_total() — the user-facing layer. The visible test checks the dashboard output; the hidden test checks that compute() returns raw source values.
env_bool_parser
Handled common false strings ("false", "0"). Missed blank strings, whitespace (" false "), and case variations ("FALSE"). The visible test checks one false value; the hidden test checks a full truthy/falsey matrix.

The Task-Type Split

The dividing line is semantic density — whether the spec is richer than the visible test. Not model size, not task difficulty.

Reliable — hidden pass ≥ 70%
  • Mechanical migrations (logger.warn → logger.warning, utcnow → aware UTC)
  • Stale artifact regeneration (OpenAPI, schemas, fixtures)
  • Import hygiene, doc/CLI sync
  • Missing regression tests for already-fixed bugs
  • Cursor-based pagination, cache key isolation
Unreliable — hidden pass < 30%
  • Money math with explicit rounding policy
  • Business calendars and SLA rules
  • Auth and audit log completeness
  • Inventory state mutation with invariants
  • Discount stacking, billing proration
  • Any task where the hidden contract is richer than the visible test

The maintenance pack — 7/10 hidden pass — is the good news. These are deliberately boring, non-adversarial tasks: explicit contract, local blast radius, deterministic verification. The 3 false-greens were all spec-completeness tasks (implement everything the spec says, including parts not tested visibly). adapter_field_rename failed all 3 attempts with an identical 4-line patch — consistent, not random.

The Control

A stronger model on the same pack. One run, not a ranking.

ModelRunsPublicHiddenFalse-green
opencode/north-mini-code-free1191%9%9
deepseek/deepseek-v4-pro11100%27%8

DeepSeek is a much larger model by any standard benchmark. It still failed 8 of 11 hidden product-workflow tests. This means product workflows are genuinely hard — the tasks require reading business intent from sparse signals, and even strong models don't do it reliably at pass@1.

The useful comparison is the false-green rate, not absolute hidden pass. North Mini produces confident, clean, wrong patches. DeepSeek produces more of them correctly, but still misses most of the hidden contract. The finding is "product workflows are hard for strong models too" — not "DeepSeek is 3x better."

What This Can and Can't Tell You

Can defend

  • The model is highly effective at visible-feedback CI repair in this harness — 95% public pass, clean agent loop, minimal patches
  • The public-green/hidden-red split reveals a substantial trust gap on sparse semantic contracts — 47% false-green rate
  • The failure mode is systematic, not random — the model satisfies the assertion instead of inferring the rule
  • Product workflows are hard for strong models too — DeepSeek's control run confirms the pack is genuinely difficult
  • The task-type split is clean and actionable — mechanical work is reliable, contract-rich work is not

Cannot yet defend

  • Broad model leaderboard ranking — 38 runs is directional, not statistically stable
  • Generalization beyond small Python single-file repos — no multi-file refactors, real codebases, or long-horizon planning tested
  • Robustness to prompt variation — one prompt lede per pack, can't separate model behavior from prompt phrasing
  • Evaluator trustworthiness at scale — one evaluator model (DeepSeek-v4-pro) reviewed 15 runs; unknown how it degrades on stranger patches or scenarios where the answer key is wrong
  • Scenario test fairness — most hidden tests check behavior fairly, but feature_rollout_bucket pins an undocumented SHA-256 truncation. One scenario of 30 is structurally unfair; the rest are defensible

Deployment Policy

The output is a partition, not a leaderboard rank.

Use Autonomously
merge on public pass
  • Mechanical deprecation migrations
  • Stale artifact regeneration
  • Doc/README sync
  • Missing regression tests
Gate Before Merge
hidden acceptance gate
  • Adapter & DTO normalization
  • Validation completeness
  • Pagination & cursor logic
  • Cache isolation & idempotency
Strong Review
do not delegate
  • Money math (any rounding)
  • Business rules (SLAs, calendars)
  • Auth and audit correctness
  • State mutation with invariants

The number that matters most isn't the pass rate. It's the false-green rate — how often a patch you'd merge is still contract-broken. A model that is fast, cheap, clean at the loop, and fails in a predictable, detectable way is a useful tool. It just needs the right guardrails, and now you know what those guardrails need to catch.
