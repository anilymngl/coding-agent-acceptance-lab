The Trust Gap

When a coding agent makes your CI green, how often is the bug actually fixed? We tested three very different models through the same harness, same prompts, same hidden acceptance tests.

The answer is the same for all three: it depends entirely on what you're asking it to do.

Question
02

EVERY BENCHMARKMARK ASKS THE WRONG QUESTION

Benchmarks measure whether
the test passed.
We measure whether the bug was fixed.

When you reward an agent for making tests green, it optimizes for the visible assertion. It stops when CI passes. It doesn't ask "what else should be true here?" — because nobody told it there was more to check.

Real fix

Public test passes. Hidden acceptance test passes. The agent understood the problem and the contract behind it.

False-green

Public test passes. Hidden acceptance test fails. The agent satisfied the visible assertion without fixing the underlying contract. CI lied.

Public red

The agent couldn't make CI green at all. Nothing to review. This is honest failure — at least you know it didn't work.

The trust gap = public pass rate − hidden pass rate.

It measures how often "looks fixed" is actually "is fixed."

Method
03

HOW THE HARNESS WORKS

Visible tests during the work.
Hidden tests after the agent exits.

1

Write a broken repo with a failing test.

The agent sees the CI log, the source code, and the public test. Exactly what a real engineer would see.

2

Let the agent fix it through OpenCode.

Same CLI, same agent loop, same tool surface for all three models. Read, edit, test, exit.

3

Run the public tests. Record CI state.

This is what every benchmark measures. This is where every benchmark stops.

4

Inject hidden acceptance tests. Run them.

Edge cases. Backward compatibility. Business rules. The agent never saw these. They test whether the fix is real or just visible.

Two prompt lanes

Sparse (control)

"CI is red. Fix it." The agent must infer what "done" means from the failing test alone.

Contract-visible (treatment)

Full acceptance criteria in the prompt. Expected behavior. Success signals. Known failure modes. The agent knows the rule.

Outcome taxonomy

hidden_pass
real fix
false_green
CI lied
public_red
honest fail
timeout
infra, not model

Runtime failures are classified separately. A timeout is not a wrong answer.

Pattern
06

WHAT FALSE-GREENS ACTUALLY LOOK LIKE

The agent asks "what makes this test pass?"
An engineer asks "what else should be true?"

decimal_money_rounding

Test: 0.29 × 3 = 87¢. Agent uses float math. Returns 87. Green.

Hidden: 0.005 × 1 with half-up rounding should be 1¢. Returns 0. Silent billing error on every half-cent transaction.

audit_log_redaction

Test: redact password field. Agent replaces top-level key. Green.

Hidden: nested metadata.api_key, profile.token, items[0].authorization. None redacted. Didn't recurse.

metric_semantic_mismatch

Dashboard combines cents + dollars. Agent normalizes inside compute() — the raw data API. Green.

Hidden: compute() must return raw values. Normalization belongs in dashboard_total(). Right math. Wrong layer.

inventory_reservation_idempotency

Test: same idempotency key shouldn't double-reserve. Agent caches the key. Green.

Hidden: insufficient stock should reject without mutating state. Stock decrements on failed reservation. Inventory goes negative.

The consistent failure direction: the agent satisfies the specific failing assertion instead of inferring the rule the assertion was testing. It gets the number right for the one value the test checked. Wrong rounding mode. Wrong API layer. Wrong cache key. This is not random. It is learnable. Guardrails can be built around it.

Insight
07

CONTRACT-VISIBLE IS THE TREATMENT. SPARSE IS THE CONTROL.

The gap between sparse and contract-visible
tells you what kind of problem this is.

North Mini: the contract reclassifies existing greens

North Mini already gets CI green on almost everything. The contract doesn't create new greens — it turns false-greens into real fixes. ci_forensics: 4 false-greens → 1. Maintenance: 2 → 0. The model stops too soon under sparse. Give it the rule, it delivers.

Gemma 12B: the contract acts as scaffolding

Gemma often can't close CI at all under sparse. The contract helps it produce a working patch where it couldn't before. ci_forensics: 7 public-reds → 5. Hidden pass: 3 → 6. The model has the capability. It needs the spec to unlock it.

DeepSeek: the contract reveals the ceiling

DeepSeek with clear specs hits zero false-greens on bugs and maintenance. 22 of 22 real fixes. But product workflows still show 9% false-green even with clear specs. The frontier model's ceiling is the spec density, not the model size.

Product workflows: the wall everyone hits

Under sparse: 73–89% false-green across all three models. Under contract-visible: 9–33%. The improvement is real but 9% on money logic is not zero. One scenario — bulk_invite_dedupe — survives even DeepSeek with clear specs.

The deception is not in the model. It is in the gap between what the test checks and what the contract requires.

Deploy
08

WHAT WE CAN AND CANNOT CLAIM

Honest caveats

pass@1. One attempt per task. Gemma's maintenance hidden pass swung 4/10 → 7/10 across two same-day runs. Single attempts describe an attempt, not a model.

System comparison. North Mini runs on its native tool surface. Gemma runs through a foreign agent loop on local Ollama. Speed asymmetry (25s vs 240s vs 37s) confounds raw capability.

33 scenarios. One evaluator. No external audit. First-stage evidence. The method is what matters, not the specific numbers.

Mixed sources. DeepSeek product sparse from a prior control run (control-results.sqlite). Other 17 cells from the matrix pipeline. Same harness, same scenarios, same hidden tests.

Deployment zones

Green — merge on CI green

Mechanical work. Migrations. Fixtures. Docs. Imports. All three models converge at 0–14% false-green with clear specs. DeepSeek: 22/22 real fixes across bugs and maintenance.

Amber — gate on hidden acceptance

Adapters. Validators. Edge cases. Field renames. Every model produced false-greens here under sparse. Clear specs close the gap. Run a hidden check before merge.

Red — human review or frontier model with multiple attempts

Billing. Discounts. Inventory. SLAs. 73–89% false-green under sparse. 9–33% even with clear specs. The policy density is too high for pass@1.
