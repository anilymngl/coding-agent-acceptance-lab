The Trust Gap: Measuring Acceptance Reliability in Coding-Agent CI Repair

Anil Yamangil

Independent Research · June 2026

Abstract

We present a behavioral evaluation of five coding-agent models across 33 curated CI-repair scenarios with hidden acceptance tests injected after the agent exits. Models achieve 55–100% public pass rates but 27–68% hidden pass rates, producing a consistent trust gap of 20–62 percentage points. The trust gap persists across model families (Cohere North Mini, Google Gemma 4, Poolside Laguna XS.2, DeepSeek V4 Pro), model sizes (3B to frontier), and deployment contexts (local Ollama, hosted API, free tier). Three scenarios remain unconquered by any model at pass@1. Contract-visible prompts collapse the gap to 0–8% on mechanical tasks but only to 9–33% on business-logic tasks. We find that pass@3 retries do not converge — North Mini regressed from pass@1 to pass@2, and both models failed all 9 retry attempts on the 3 unconquered scenarios. We identify a secondary trust gap: the model's own post-repair summary declares success in 3 of 4 false-green cases. We release the harness, all artifacts, and evaluation methodology as open-source evidence.

Keywords: coding agents, CI repair, trust gap, hidden acceptance, false-green, evaluation harness

1. Introduction

Standard coding benchmarks measure whether a model can generate code that passes tests. This evaluation asks a different question: when a coding agent makes CI green, how often is the bug actually fixed?

The distinction matters because visible tests create an optimization attractor. The agent optimizes for the public test signal — the only feedback available during its run. It stops when CI passes. It does not ask "what else should be true here?" because nobody told it there was more to check.[1]

A model that satisfies the visible assertion without satisfying the hidden contract is a false-green: public_pass=1, hidden_pass=0. The trust gap — public pass rate minus hidden pass rate — measures how often "looks fixed" is actually "is fixed."

We evaluate five models across three task types with two prompt lanes (sparse and contract-visible) at pass@1. The evaluation produces three findings:

1. The trust gap is structural, not model-specific. Two independently trained 3B-active MoE coding agents from different companies produce identical false-green scenarios.

2. Contract-visible prompts collapse the gap on mechanical tasks but not on business-logic tasks. The frontier model's ceiling is the spec density, not the model size.

3. The model's own post-repair summary declares success in 3 of 4 false-green cases, creating a secondary trust gap between the model's explanation and reality.

2. Methodology

2.1 Harness Design

Each scenario is a self-contained Python repository with a failing public test suite. The harness executes the following pipeline:[2]

1. Write the scenario to a fresh worktree.

2. Run baseline tests (confirm red CI).

3. Invoke the coding agent via OpenCode CLI with the rendered prompt.

4. Run public tests (record CI state).

5. Inject hidden acceptance tests into the worktree.

6. Run hidden tests (record acceptance state).

7. Store all artifacts: prompt, stdout JSONL, patch, test outputs.

Runtime failures (timeouts, provider errors) are classified separately from semantic outcomes. A no_output_timeout is operational evidence, not a model failure.[3]

2.5 Evaluator Agent

False-green rows are reviewed by a separate evaluator agent (DeepSeek V4 Pro) with a workbench: the original repo, the model's patch, the hidden tests, and a blank shadow_repo for writing alternative fixes. Every claim in the evaluator's verdict must appear verbatim in the evidence packet. Pydantic enforces schema compliance. The evaluator cannot hallucinate its way to confidence.[4]

3.7 The Deterministic False-Green

decimal_money_rounding produced byte-identical patches on Laguna attempts 1 and 2:

- total = 0.0
+ total = Decimal("0")
- total += float(line["unit_price"]) * int(line["quantity"])
+ total += Decimal(line["unit_price"]) * int(line["quantity"])
  return int(total * 100)  ← truncates, doesn't round half-up

The model independently produced the same wrong code from scratch, twice. This is not random variance. It is a stable attractor — the model converges to the same incomplete solution when the public test signal does not push it toward the hidden contract.

4. Discussion

4.1 The Trust Gap Is Structural

Five models across four model families produce the same pattern. Two 3B-active MoE coding agents from different companies (North Mini, Laguna) hit identical false-green rates on identical scenarios. The frontier model (DeepSeek) is only 1 scenario better at pass@1. The trust gap is not a function of model size, training data, or reasoning capability. It is a function of the gap between the visible test and the hidden contract.

4.2 The Contract-Visible Effect

Contract-visible prompts collapse the trust gap on mechanical and refactoring tasks (0–14% false-green) but only reduce it on business-logic tasks (9–33%). The intervention is the specification, not the model. This is consistent with the challenge design philosophy: the hidden contract is richer than the visible test, and the model cannot infer the full contract from partial evidence alone.[1]

4.3 The Spec Density Ceiling

DeepSeek with contract-visible prompts reaches 0% on ci_forensics and maintenance_value but only 9% on product_workflows. The frontier model's ceiling is the spec density — the ratio of hidden contract to visible test. Business-logic scenarios encode policy (billing rules, audit contracts, inventory state machines) that cannot be fully specified in a prompt without revealing the hidden tests.

4.4 Retries Do Not Converge

Laguna's pass@3 data shows independent fresh attempts converge on the same incomplete solutions — decimal_money_rounding produced identical patches twice. North Mini's pass@3 data is more concerning: the model regressed from pass@1 to pass@2, losing a previously solved scenario without gaining new ones. Across both models, 9 independent attempts on the 3 unconquered scenarios produced zero successful escapes. The public test signal is too weak to guide either model out of its attractor. Re-sampling from a distribution of wrong answers can produce worse outcomes, not better ones.

4.5 Implications for Deployment

The evidence supports a three-zone deployment policy:

Green Merge on CI green. Mechanical work. Migrations. Fixtures. Docs. All models converge at 0–14% false-green with clear specs.

Amber Gate on hidden acceptance. Adapters. Validators. Edge cases. Clear specs close the gap but do not eliminate it.

Red Human review or frontier model with clear specs. Billing. Discounts. Inventory. SLAs. Auth. Audit. 67–89% false-green under sparse. 9–33% even with clear specs.

The model is replaceable. The trust gate is the enduring investment.

5. Limitations

1. Pass@1 is noisy. Gemma 12B's maintenance hidden pass swung from 4/10 to 7/10 across two same-day runs. Single-attempt numbers describe an attempt, not a model. Pass@3 minimum is needed for defensible claims.

2. System comparison, not model ranking. North Mini runs on its native tool surface. Gemma runs through a foreign agent loop on local Ollama. Speed asymmetry (17s vs 211s) confounds raw capability. We measure system behavior, not abstract model quality.

3. 33 scenarios. One evaluator. No external audit. The hidden tests and evaluator verdicts are authored by the same team. Tier 1 evidence — behavior microscope only. Tier 2 (comparative claims) needs evaluator agreement. Tier 3 (public benchmark) needs external audit.

4. Incomplete matrix. Four cells complete (ci_forensics sparse + cv for both models, 144 runs). Maintenance and product_workflows cells still running. DeepSeek product_workflows sparse is from a prior control run. Conclusions strengthen with more data.

6. Related Work

SWE-bench[5] measures resolution accuracy on real GitHub issues. Terminal-Bench[6] measures agentic task completion. This harness measures a different axis: whether public CI green can be trusted as acceptance when the visible tests under-specify the product contract. The public/hidden split is the methodological contribution — not a new benchmark, but a trust audit on top of existing agent capabilities.

7. Conclusion

The trust gap is real, structural, and cross-model. When a coding agent makes CI green, there is a 20–62% chance the bug is not actually fixed, depending on task type and prompt specificity. The gap persists across five models from four families. It collapses to 0–14% with explicit acceptance criteria on mechanical tasks but remains at 9–33% on business-logic tasks even for frontier models.

The practical implication is clear: partition your CI repair by task type, not by model. Mechanical work can be delegated with clear specs. Business logic needs hidden acceptance gates. The model is replaceable. The trust gate is not.
