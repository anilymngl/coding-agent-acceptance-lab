Pass@3 Trust Gap: Two Coding Agents, 33 Scenarios, and a Wall They Cannot Climb

Anil Yamangil

Independent Research · June 2026 · v2 — pass@3 matrix update · research suite

Home | Paper | Harness | Catalog | Evidence | Evaluator

Abstract

We run a head-to-head pass@3 evaluation of two independently trained 3B-active MoE coding agents — Cohere North Mini and Poolside Laguna XS.2 — across 33 CI-repair scenarios with hidden acceptance tests injected after the agent exits. On ci_forensics sparse, both models converge to identical 75% hidden pass rates (9/12 scenarios) at best-of-3, with two scenarios unconquered by both models (0 hidden passes across all 6 attempts each). Contract-visible prompts collapse the false-green rate from 25–30% to 0–9% across all packs for both models. Laguna reaches 100% hidden pass on maintenance_value and product_workflows contract-visible — zero false-greens. Product workflows sparse produces 64–82% false-green rates with 7 of 11 scenarios completely unconquered at pass@3 by both models. Retries do not converge: North Mini regressed from pass@1 to pass@2, losing a solved scenario without gaining any. Laguna leads by 3 hidden passes overall (54/66 vs 51/66 best-of-3), with the entire gap in product_workflows. The evidence supports delegating mechanical and refactoring tasks with explicit specs while requiring hidden acceptance gates for business logic. The model choice matters less than the task type.

Keywords: coding agents, CI repair, trust gap, false-green, hidden acceptance, pass@3, evaluation harness, contract-visible

1. Introduction

Standard coding benchmarks measure whether a model can generate code that passes tests. This evaluation asks a different question: when a coding agent makes CI green, how often is the bug actually fixed? Observed 391 non-timeout runs across 2 models, 3 packs, 2 prompt modes, 3 attempts each

The distinction matters because visible tests create an optimization attractor. The agent optimizes for the public test signal — the only feedback available during its run. It stops when CI passes. It does not ask "what else should be true here?" because nobody told it there was more to check. A model that satisfies the visible assertion without satisfying the hidden contract is a false-green: public_pass=1, hidden_pass=0. The trust gap — public pass rate minus hidden pass rate — measures how often "looks fixed" is actually "is fixed."

This paper is the v2 update of the original Trust Gap report, now with pass@3 data — three independent fresh starts per scenario with no memory of prior attempts. The original v1 paper established the core finding across 5 models from 4 families: a consistent 20–62 point trust gap at pass@1, with three scenarios unconquered by any model. v2 narrows to a head-to-head pass@3 comparison of two models with identical architecture (3B-active MoE) but different training data and deployment — Laguna XS.2 (Poolside, OpenRouter free) and North Mini (Cohere, OpenCode free). Every scenario, prompt, and hidden test is identical. Every run was automated through the same harness.

1. The two models are indistinguishable on ci_forensics and maintenance_value at pass@3 best-of-3. Both converge to identical hidden pass rates.

2. Contract-visible prompts collapse the trust gap from 25–30% to 0–9% across all packs for both models. Laguna reaches 0% on maintenance_value and product_workflows cv.

3. Retries do not converge. North Mini regressed from pass@1 to pass@2 — re-sampling from a distribution of wrong answers can produce worse outcomes.

4. Product workflows sparse is the trust killer: 64–82% false-green, 7 of 11 scenarios unconquered by both models. Contract-visible recovers most but not all.

5. Laguna leads by 3 hidden passes overall (54/66 vs 51/66), with the entire gap in product_workflows.

2. Method

2.1 Harness

Each scenario is a self-contained Python repository with a failing public test suite. The harness executes: (1) write scenario to fresh worktree, (2) run baseline tests to confirm red CI, (3) invoke the coding agent via OpenCode CLI with the rendered prompt, (4) run public tests to record CI state, (5) inject hidden acceptance tests, (6) run hidden tests, (7) store all artifacts. Each pass@k attempt is an independent fresh start — new worktree, no memory of prior attempts.[1]

On sparse, 7 of 11 scenarios are completely unconquered by both models — 0 hidden passes across 3 fresh attempts each. The model can always make CI green. It almost never makes the product correct. The public test signal is too sparse to guide either model toward the hidden business contract.

Contract-visible is not a timeout wall — earlier runs hit free-tier quota exhaustion before the lane could complete, creating a false impression that heavier prompts stalled the agent. 3.6 The Deterministic False-Green: decimal_money_rounding

Laguna produced byte-identical wrong patches on all 3 attempts of decimal_money_rounding:

- total = 0.0
+ total = Decimal("0")
- total += float(line["unit_price"]) * int(line["quantity"])
+ total += Decimal(line["unit_price"]) * int(line["quantity"])
  return int(total * 100)  ← truncates toward zero, doesn't round half-up

The model independently produced the same wrong solution three times, each from a fresh worktree with no memory. This is not random variance. It is a stable attractor — the model converges to the same incomplete solution because the public test does not push it toward the hidden contract. The evaluator agent (DeepSeek V4 Pro) confirmed: int(Decimal("0.005") * 100) = int(Decimal("0.500")) = 0, truncating instead of rounding half-up to 1. Observed Evaluator verified, mean confidence 0.95, root cause = edge_case_gap

False-greens are not smaller patches — they are slightly larger (7.4–7.7 changed lines vs 6.0–6.1 for hidden passes). Tool-call counts tell a different story. False-greens use fewer tool calls than hidden passes (10.7–10.8 vs 13.1–14.2 average). The model stops sooner on false-greens — it reaches the visible-test-satisfied state and exits. Hidden passes require more exploration: more reads, more bash commands, more edits. 4. Discussion

4.1 The Trust Gap Is Structural, Not Model-Specific

Two independently trained 3B-active MoE coding agents from different companies produce nearly identical false-green rates on identical scenarios. On ci_forensics sparse, both solve 9 of 12 scenarios and fail the same 3. On maintenance_value sparse, both solve 8 of 10. On ci_forensics and maintenance_value contract-visible, both reach 0% false-greens at best-of-3. The trust gap is a function of the gap between the visible test and the hidden contract — not of model size, training data, or reasoning capability. The only divergence is product_workflows, where Laguna leads by 3 hidden passes.

Explicit acceptance criteria collapse the trust gap to 0% on mechanical and refactoring tasks for both models. On product_workflows, the effect is large but not complete: Laguna reaches 0% with specs, North Mini retains a 9.1% residual (1 false-green on support_sla_business_hours across all 3 attempts). The intervention is the specification, not the model. When the agent knows what acceptance looks like, it can usually reach it — even on business logic.

4.3 The Residual Wall: Business Logic Under Sparse Prompts

Under sparse prompts, product_workflows is a near-total failure: 64–82% false-green, 7 of 11 scenarios completely unconquered by both models at pass@3. The public test signal is too weak to guide either model toward the hidden business contract. But unlike the earlier interpretation (when the contract-visible lane had not yet completed), we now know that explicit specs do recover most product-workflow scenarios — Laguna reaches 11/11, North Mini reaches 10/11. The wall is not that business logic cannot be specified; it is that the model cannot infer the specification from a failing test alone. When the spec is provided, the model can implement it. The residual false-green on support_sla_business_hours for North Mini suggests some business policies remain ambiguous even with explicit acceptance criteria.

4.4 Implications for Deployment

The evidence supports a three-zone deployment policy based on task type, not model choice:

Green Delegate with clear specs. Mechanical tasks (header contracts, parsing, rounding) and refactoring (migrations, renames, schemas). Contract-visible collapses trust gap to 0%. Both models perform identically.

Amber Delegate with specs + hidden acceptance gate. Business logic (billing, discounts, inventory, SLAs). Contract-visible collapses the gap from 64–82% to 0–9%. Laguna reaches 0%; North Mini retains 1 residual false-green. Run hidden acceptance tests before merge.

Red Do not delegate without specs. Business logic under sparse prompts. 64–82% false-green, 7 of 11 scenarios completely unconquered at pass@3 by both models. The model can make CI green but almost never makes the product correct. Human review or explicit acceptance criteria are mandatory.

5. Limitations

1. Near-complete matrix. 391 of 396 runs completed (98.5%). 5 runs missing on North Mini maintenance_value contract-visible (OpenCode Zen quota exhaustion, pending retry). Cross-model claims are supported across all 3 packs. Not built 5 of 30 runs in 1 cell

2. Gateway switch for North Mini. North Mini was accessed through two free-tier gateways: OpenCode Zen (ci_forensics, maintenance_value) and OpenRouter (product_workflows, after Zen quota exhaustion). The underlying model is the same Cohere North Mini Code. If the two gateways serve different model versions or inference configurations, we cannot separate gateway effects from model behavior. No behavioral difference was observed across the switch boundary, but it cannot be ruled out. Caveat Two gateways used for North Mini

3. Incomplete evaluator coverage. Only 10 Laguna false-greens have been reviewed by the evaluator agent (all pass@1 ci_forensics sparse). The remaining 91 false-greens are unverified. Evaluator verdicts may not be reproducible without the same evaluator model. Not built evaluator review for pass@2, pass@3, maintenance, product, North Mini

4. System comparison, not model ranking. Laguna runs on OpenRouter free tier. North Mini runs on OpenCode Zen + OpenRouter free tiers. Speed asymmetry (17s vs 43s) is confounded by deployment surface. We measure system behavior, not abstract model quality.

5. Both models share MoE architecture (3B active). The head-to-head finding that "model choice doesn't matter" may be specific to this architecture class. The v1 study (paper.html) found the same trust-gap pattern across 4 model families including dense models and a frontier model, suggesting structural rather than architectural cause. But the convergence-at-pass@3 claim is only validated for these two MoE models.

6. 33 scenarios. One harness. No external audit. The hidden tests and evaluator verdicts are authored by the same team. Tier 1 evidence — behavior microscope only. Tier 2 (comparative claims across models) needs evaluator agreement. Tier 3 (public benchmark) needs external audit.

6. Conclusion

Two independently trained 3B-active MoE coding agents from different companies produce nearly identical false-green patterns at pass@3 across 33 CI-repair scenarios. The trust gap is structural — a function of the distance between the visible test and the hidden contract, not of the model. Contract-visible prompts eliminate the gap entirely on mechanical and refactoring tasks (0% FG for both models) and nearly eliminate it on business logic (0% for Laguna, 9.1% for North Mini). The earlier finding that product_workflows contract-visible was a "timeout wall" was an artifact of free-tier quota exhaustion, not a model capability ceiling.

Retries do not converge. North Mini regressed from pass@1 to pass@2. The marginal gain from retry on ci_forensics is +1 hidden pass for Laguna and +0 for North Mini. Three scenarios remain unconquered across 18 independent attempts on ci_forensics sparse. Product workflows sparse is worse: 7 of 11 scenarios unconquered across 6 attempts each by both models.

Laguna leads by 3 hidden passes overall (54/66 vs 51/66 best-of-3), with the entire gap in product_workflows (4 vs 2 on sparse, 11 vs 10 on cv). The models are indistinguishable on ci_forensics and maintenance_value. The practical implication: partition CI repair by task type, not by model. Mechanical and refactoring work can be delegated with clear specs. Business logic needs explicit acceptance criteria and hidden-test gates. The model is replaceable. The trust gate is not.
