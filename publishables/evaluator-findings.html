How reviews work

What is a false-green? The model's patch made the public test pass (CI green) but failed hidden tests the model never saw. The bug wasn't actually fixed.
Evaluator model A separate AI (DeepSeek V4 Pro) that never saw the original run. It receives only: the trap description, what the correct behavior should be, the model's code diff, and the hidden test output. It cannot see the scenario source code.
Shadow fix The evaluator must write its own working fix in a blank repo. This proves the correct approach exists — the hidden tests aren't broken. If the evaluator can fix it, the original model's failure was real.
Evidence gate Every claim in the verdict must be an exact quote from the evaluation packet. The schema forbids extra fields (extra="forbid"). The evaluator cannot make things up.
Root cause categories:
Edge gap The model understood the rule but missed specific boundary cases (e.g. empty strings, whitespace, zero values).
Missed contract The model didn't know the hidden requirement existed. Nothing in the visible test signaled this was required.
Incomplete policy The model understood the direction but didn't implement the full rule (e.g. changed the type but kept wrong rounding, or fixed one file but missed another).
Patch quality 1-5 scale. 1 = barely relevant. 3 = correct direction, incomplete. 5 = full fix.
Debug discipline 1-5 scale. Did the model investigate before editing? 1 = edited blindly. 5 = read multiple files, traced the data flow.
