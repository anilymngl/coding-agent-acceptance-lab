Why this works: Agentic judgment is enclosed by deterministic preparation and deterministic validation. The evaluator is not a free-range reviewer. It operates inside a reproducible workbench with constrained evidence. The system does not ask “do you agree this patch is wrong?” It asks the evaluator to rebuild the failure, diagnose the violated contract, and support the diagnosis with evidence from the packet.

The harness does the boring authoritative work. The evaluator does the uncertain work.

The runner rebuilds the repository, applies the captured patch, injects the hidden tests, and preserves the packet. The evaluator inspects, reasons, diagnoses, and optionally tests an alternative implementation. Then deterministic checks decide whether the review is admissible — not whether it is correct, but whether it is structurally valid, consistent with the stored run, and supported by exact evidence from the packet.

A false-green tells us that the agent made the visible tests pass but failed hidden acceptance. That identifies the symptom. It does not explain why the patch failed. For selected false-greens, we give a separate evaluator agent a disposable engineering workbench.

The evaluator must work through three stages:

1. Reproduce the failure

The evaluator applies the original model patch and runs the same public and hidden tests. This checks whether the retained patch reproduces the recorded result (public tests pass, hidden tests fail). If the failure cannot be reproduced, the evidence is incomplete and the review should not be treated as validated.

2. Diagnose the missed contract

The evaluator inspects the code, test output, and authored acceptance contract. It must identify the specific mismatch between the patch and the required behavior. The diagnosis must point to concrete evidence from the workbench. A verdict without an observable code path, test failure, or contract mismatch is not enough.

3. Build a shadow fix (Optional)

The evaluator may edit a clean shadow repository and attempt an alternative implementation. The shadow fix is tested against the same visible and hidden suites. When the shadow fix passes, it shows that the authored contract is implementable in the supplied repository and supports the evaluator’s diagnosis of a concrete gap in the original patch. Note that shadow-fix outcomes are currently not structured in the results database.

What this establishes

A validated review means the verdict is structurally valid, consistent with the stored run, and supported by exact evidence from the packet. It does not mean the evaluator has discovered the one true explanation. The narrow claim it supports:

  • Under the authored benchmark contract, the original agent patch was incomplete or incorrect, and the evaluator demonstrated an alternative implementation that satisfied the same acceptance tests.

What this does not establish

The evaluator does not independently prove that:

  • the hidden tests are complete;
  • the scenario is fair or representative;
  • the benchmark contract matches every real-world requirement;
  • the evaluator’s root-cause category is uniquely correct.

The workbench makes the diagnosis executable and auditable. It does not turn one evaluator model into an external scientific authority. Another model's opinion is not evidence. The workbench is what makes the diagnosis inspectable.
