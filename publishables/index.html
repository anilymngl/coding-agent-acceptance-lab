A coding agent can make CI green without fixing the bug.

A green test suite proves that the visible checks passed. It does not always prove that the underlying business rule, policy, or acceptance contract was implemented correctly.

This project measures that gap across four model families and a 391-attempt depth study. The question is whether a coding-agent patch is actually accepted, not only whether it made public CI pass.

CI: automated checks that say whether a code change appears safe.
Public tests: checks the agent can see and run while fixing the repo.
Hidden acceptance: checks injected after the agent exits.
False-green: public tests pass, hidden acceptance fails.
Acceptance contract: the behavior or policy the fix actually has to satisfy.
Trust gap: public pass rate minus hidden pass rate.

What the study found

01

Green is only the first gate

Public tests tell us the patch satisfied the visible assertions. In the depth study, 101 public-green attempts still failed the broader authored contract.

02

The contract mattered more than another roll of the dice

Contract-visible prompting reduced the pooled false-green rate from 43.3% to 8.9%. Individual reruns were not progressively better.

Three attempts are not three layers of wisdom.
03

Product logic remained the hardest edge

Billing, inventory, security, ranking, and SLA workflows remained harder because one visible failure often exposes only one edge of a larger policy.

Evidence boundary

The primary depth study contains 391 retained attempts across two deployed systems, two prompt lanes, and 33 authored scenarios.

The breadth study and supporting Gemma experiments use different configurations and scopes. They are reported separately rather than pooled into one model leaderboard.

The hidden tests encode the authored benchmark contracts. They are stronger than the visible tests, but they are not universal proof of production correctness.
