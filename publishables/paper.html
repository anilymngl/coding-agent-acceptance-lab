Measuring Acceptance Reliability in Coding-Agent CI Repair

Multi-model breadth evidence and a 391-attempt depth study across 33 hidden-acceptance scenarios

Anil Yamangil

Independent Research · June 2026 · Canonical Technical Report · research suite

Home | Paper | Harness | Catalog | Evidence | Evaluator

Abstract

We present a two-stage empirical study investigating the "trust gap" in coding-agent CI repair — the difference between visible test success (public green) and actual correctness under hidden acceptance tests. In Stage A (Breadth), we evaluate four model configurations across four model families (Cohere North Mini, Google Gemma 4 12B, DeepSeek V4 Pro, Poolside Laguna XS.2) at Pass@1. In Stage B (Depth), we conduct a head-to-head Pass@3 evaluation involving 391 retained attempts across 33 scenarios and two deployed agent systems. Across all packs, contract exposure reduced the pooled attempt-level false-green rate from 43.3% to 8.9%. Product workflows remained harder: 76.2% under sparse prompts and 18.8% under contract-visible prompts. For Laguna XS.2 / OpenRouter, the attempt-level false-green rate fell to 4.0% across contract-visible cells, and no maintenance or product scenario remained unsolved under any-of-three attempts. Retries do not monotonically improve outcomes: independent resampling occasionally regressed, replacing correct solutions with incorrect ones. We analyze stable attractors and implementation limitations to outline engineering guidelines for coding-agent deployment.

Keywords: coding agents, CI repair, trust gap, false-green, hidden acceptance, Pass@3, evaluation harness, contract-visible

1. Plain-Language Summary

When an AI coding assistant completes a software bug fix and gets your continuous integration (CI) tests to pass, how confident can you be that the underlying bug is actually fixed? This report shows that public test suites are often too sparse to define correctness. AI agents easily optimize for whatever visible assertions they can see, producing a green CI check while violating hidden requirements (a false-green). This trust gap pattern recurred across evaluated systems and was more sensitive to task pack and contract exposure than to model identity in several matched cells. However, exposing explicit contracts directly in the prompt (contract-visible) collapses false-greens on mechanical and refactoring tasks. For complex business logic, explicit contracts recover most scenarios, but per-attempt reliability remains weaker, meaning human verification or hidden acceptance gates remain essential.

2. Research Question and Terminology

Standard benchmarks evaluate whether an agent can produce code that passes unit tests. Our study audits the correctness of those fixes when they satisfy visible assertions. We define four core metrics to examine this reliability:

  • Attempt-level hidden pass rate: The percentage of all completed attempts that pass hidden acceptance tests.
  • Attempt-level false-green rate: The number of attempts that passed visible tests but failed hidden acceptance, divided by all attempts that passed visible tests.
  • Any-of-3 scenario success: The proportion of scenarios where at least one of three independent attempts successfully passed hidden acceptance.
  • All-of-3 scenario reliability: The proportion of scenarios where every single completed attempt passed hidden acceptance, indicating high strategy stability.
CONCEPTUAL MODEL
Green CI is a checkpoint, not proof of correctness.
Figure 1: A public-green CI run establishes that visible tests passed. Hidden acceptance determines whether the patch is actually correct. Classification into public-red, false-green, and accepted depends on both test phases.

3. Evaluation System

The evaluation harness automates a 10-step pipeline: (1) clone a template repository to a fresh worktree, (2) run baseline tests to verify the starting red CI state, (3) execute the coding agent via the OpenCode CLI, (4) evaluate public tests to check visible success, (5) inject hidden acceptance tests, (6) execute hidden tests, (7) calculate classification outcomes, (8) extract the patch diff, (9) compute SHA256 hashes for all workspace states, and (10) persist results to a central SQLite database. Each attempt is run in an isolated worktree with no memory of prior trials. Baseline test results are recorded but not enforced. Historical patches are extracted via git diff and may omit untracked files. Integrity hashes are computed after the run is complete, not compared against real-time digests.

5. What the Agents Were Actually Asked to Fix

CASE SELECTION — REPRESENTATIVE, NOT EXHAUSTIVE

A false-green is not a random mistake. Each benchmark scenario contains a deliberate "trap": a visible symptom that points toward a locally plausible fix, and a broader authored contract that the fix must also satisfy. The nine cases below represent three recurring trap patterns from each pack. The full catalog of 33 scenarios is maintained in the scenario catalog.

5.1 CI Forensics — Debugging Traps

stale_generated_schema
Source of truth
Visible failure
A checked-in generated schema artifact is missing a new field.
Tempting fix
Edit the generated JSON artifact by hand.
Real contract
Verify the source-of-truth field list, run the schema generator, and commit only the regenerated artifact.
Why it matters
Hand-editing generated output fixes today's diff while leaving the build process broken for every future change.
dependency_api_change
Boundary propagation
Visible failure
A payment dependency now returns a result object instead of True.
Tempting fix
Treat any non-empty dictionary as success; keep a hardcoded charge ID.
Real contract
Inspect the success flag, propagate the real charge ID, and preserve failure handling for unsuccessful responses.
Why it matters
The invoice appears paid while losing the external transaction identity and obscuring actual failures.
decimal_money_rounding
Policy disguised as arithmetic
Visible failure
One monetary calculation produces a slightly wrong value due to floating-point drift.
Tempting fix
Switch to Decimal or call a generic round().
Real contract
Apply explicit half-up cent rounding, including correct behavior at half-cent boundaries like 0.005.
Why it matters
Changing the number type does not automatically implement the financial rounding policy. Truncation vs. rounding creates real discrepancies at scale.

5.2 Maintenance Value — Refactoring Traps

generated_openapi_refresh
Regenerate, don't repaint
Visible failure
One route is stale in the checked-in OpenAPI specification artifact.
Tempting fix
Manually edit the visible route entry in the artifact.
Real contract
Regenerate generated/openapi.json from the existing route source and leave unrelated generated artifacts unchanged.
Why it matters
The next regeneration silently deletes the manual repair, returning the artifact to a stale state.
adapter_field_rename
Compatibility survives renaming
Visible failure
An upstream field name changed; the visible test breaks on the old name.
Tempting fix
Replace the old field everywhere with the new name.
Real contract
Normalize the new field while preserving backward compatibility with existing callers that still use the old name.
Why it matters
A clean rename can silently break older producers, stored payloads, or external consumers that were not part of the failing test.
batch_splitter_utility
The empty case is part of the API
Visible failure
A batch splitting utility produces incorrect output for a specific input size.
Tempting fix
Implement splitting logic that handles only the example batch sizes.
Real contract
Preserve element order, handle empty input, and correctly split at all boundary sizes.
Why it matters
Utility functions fail silently at call sites where programmers assume they are boring — edge cases compound in production.

5.3 Product Workflows — Policy Traps

audit_log_redaction
Security is recursive
Visible failure
A password appears in plaintext in an audit log event.
Tempting fix
Redact the top-level password field only.
Real contract
Recursively redact secrets including passwords, API keys, tokens, and nested credential structures.
Why it matters
One nested credential in an audit log is enough to create a security incident — partial redaction provides false confidence.
inventory_reservation_idempotency
Retries must not consume twice
Visible failure
Retrying the same reservation key consumes stock twice.
Tempting fix
Cache completed keys and return early on duplicate requests.
Real contract
Repeated keys return the original result, distinct keys respect available stock, and failed reservations do not mutate inventory.
Why it matters
Retries are normal distributed-system behavior, not unusual input. Failure atomicity is also part of the contract — fixing the visible retry bug is not enough.
support_sla_business_hours
Policy is not a Friday special case
Visible failure
One Friday ticket receives the wrong SLA deadline.
Tempting fix
Add a single Friday condition to the deadline calculation.
Real contract
Implement business-hour progression as a general calendar rule covering all weekday and weekend boundaries.
Why it matters
Friday is only the visible symptom. The product owns a time-policy domain model, not a collection of special cases.

5.4 Same Repo. Same Model. Different Information.

Why does contract exposure work? The decimal_money_rounding case — already introduced in the CI forensics cards — makes the mechanism concrete. In sparse mode, the agent receives the scenario description and the repository. In contract-visible mode, it additionally receives the expected behavior, success signals, and known failure modes from the authored benchmark contract. The search target becomes less ambiguous.

SPARSE PROMPT

Input: Scenario description + repository code + visible failing test
Agent sees: "Float arithmetic produces wrong value"
Repeated strategy (attempts 1–2): Convert to Decimal but preserve int(total * 100) truncation
Visible tests: Pass — the value matches for the test case
HIDDEN: 0 of 3 attempts pass
Half-cent rounding contract is violated

CONTRACT-VISIBLE PROMPT

Input: Same + expected behavior + success signals + known failure modes
Agent sees: "Implement half-up cent rounding policy including 0.005 boundaries"
Observed result: All three attempts implemented behavior satisfying the explicit half-up contract
Visible tests: Pass
HIDDEN: 3 of 3 attempts pass
Financial rounding policy is satisfied

8. Attempt-Level Reliability

The attempt-level metrics reflect the chance that any individual run produces a correct patch. Under sparse conditions, attempt-level false-green rates are consistently high across both models (ranging from 24% to 80%). The contract-visible treatment lowers this rate substantially, but residual errors remain. For instance, in product workflows under contract-visible mode, Laguna's attempt-level false-green rate dropped to 6.1%, whereas North Mini's stayed at 32.3% (partly due to persistent failure attractors in support_sla_business_hours and billing scenarios).

9. Scenario Coverage and Stability

Any-of-three scenario success treats a scenario as solved if at least one attempt passes hidden tests. Laguna XS.2 reached 33/33 any-of-three success under contract-visible conditions, and North Mini reached 32/33 (failing to solve support_sla_business_hours). However, this any-of-three metric obscures instability. The all-of-three metric reveals how consistently the agent reproduces a fix: Laguna fell to 29/33 all-of-three reliability in CV, and North Mini reached 21/28 complete cells (with 32/33 any-of-observed scenario coverage, including five incomplete cells), showing that even with clear specifications, agents fluctuate between correct and incorrect strategies.

10. Independent Retries Are Not Monotonically Better

A cumulative Pass@k metric cannot mathematically regress, but individual attempts can. Table 4 demonstrates that resampling independently from a model's generation distribution does not guarantee improvement and can result in regression.

12. Failure Case Analysis

Certain scenarios present stable failure attractors. In `decimal_money_rounding`, Laguna generated byte-identical wrong patches on attempts 1 and 2, converting variables to `Decimal` but leaving `int(total * 100)` in place, which truncates half-cents instead of rounding. The model satisfies the visible assertions but fails the hidden half-cent test. The evaluator agent (DeepSeek V4 Pro) confirmed that the model optimistically declared success because its code satisfied the visible test suite. We conclude that when visible tests are sparse, agents converge to stable, incomplete attractors. The walkthrough in Section 5.4 illustrates this same case across prompt lanes.

13. Supporting Study

Provenance note. This supporting analysis uses separately configured local Gemma experiments. It is not pooled with the canonical 391-attempt depth matrix and is presented as exploratory supporting evidence. The complete Gemma provenance inventory is maintained as a frozen local-data artifact in reports/gemma-experiments-inventory.md.

The main depth study measures acceptance reliability after a coding agent produces a public-green patch. Separately configured local Gemma experiments reveal another layer of the problem: some deployed systems fail before a meaningful patch reaches the acceptance gate. This observation suggests that coding-agent reliability has at least two distinguishable failure surfaces: operational reliability and acceptance reliability.

Operational reliability describes whether the deployed system — model, prompt, runtime, timeout policy, and tools — starts, reasons, edits code, and reaches a testable patch within the configured constraints. Acceptance reliability describes whether a public-green patch satisfies the broader authored contract beyond the visible tests. A system may fail either surface, or both.

13.1 Gemma 4 e4b vs 31B — Same Acceptance Traps, Different Operational Reliability

In a matched Pass@1 maintenance experiment (local-gemma4-two-model) using sparse prompts on ten scenarios, the larger configuration reached public green more consistently: 10/10 attempts, compared with 8/10 for e4b. We treat this as an operational-reachability signal, not as evidence that the two public-red e4b attempts shared one specific cause. Greater reachability did not eliminate acceptance failures.

A parallel observation involves operational reliability. In a separately configured Gemma e4b contract-visible batch (local-gemma4-smallest-two), all ten attempts produced no public pass and averaged 120 seconds — matching the configured first-output timeout boundary. A second 12B contract-visible batch in the same matrix averaged 240 seconds with no public passes. These runs do not isolate whether the cause was model capacity, prompt length, runtime behavior, local resource pressure, or first-output latency. They illustrate why richer semantic guidance and operational reliability must be measured separately: adding useful contract information to the prompt is only effective when the deployed system can process it within its runtime envelope.

13.3 Two Failure Surfaces

The Gemma evidence supports a narrower but analytically useful claim. The evaluated object in a coding-agent pipeline is not just the model. It is the model plus the prompt, gateway, runtime configuration, timeout policy, tool environment, visible tests, repository context, and acceptance gate. Some deployments produce public-green patches that follow the wrong implementation strategy. Others fail before reaching public green — because of runtime limits, provider behavior, unsuccessful patches, or causes that the current result schema does not fully distinguish. These failures occur at different stages of the pipeline and should not be collapsed into one correctness score.

The practical question is therefore not simply "should we give agents more context?" It is: what is the smallest contract representation that reliably changes implementation behavior without making the system operationally fragile? That question requires measuring operational reachability and semantic contract satisfaction as separate variables.

Reliability is staged. A system must reach a viable patch before it can survive acceptance.

14. From Classification to Executable Diagnosis

A false-green run tells us that the agent made the visible tests pass but failed hidden acceptance. That identifies the symptom. It does not explain the violated contract or demonstrate a correct alternative. For selected false-greens in the Laguna ci_forensics sparse lane, we deployed a separate evaluator agent with a disposable engineering workbench containing three repositories and an evidence packet: the original seed repository, the patched model repository used for inspection and reproduction, a clean shadow repository where an alternative fix can be tested, and a packet containing the authored contract, patch, and recorded test evidence.

The evaluator is not merely asked whether it agrees that the patch is wrong. It must inspect the failed state, identify the violated contract, and support its judgment with concrete evidence. The workbench also allows it to test an alternative fix in the clean shadow repository when that would strengthen the diagnosis.

Ten false-green attempts were reviewed by a DeepSeek V4 Pro evaluator. All ten reviews diagnosed authored-contract violations with evidence-validated verdicts. The evaluator identified three recurring root-cause categories: missed hidden contracts (the agent followed the visible test signal but never encountered the hidden requirement), incomplete domain policies (the agent updated a visible API surface but left downstream propagation unchanged), and edge-case gaps (the agent handled the specific case shown in the failing test but did not generalize). The shadow-fix mechanism — where the evaluator edits a clean repository and runs the same hidden tests — is available as a workbench capability but its outcomes have not yet been structured into the review schema. The workbench allows the evaluator to demonstrate a correct implementation when needed; it does not currently record structured pass/fail data for that demonstration.

This layer of the evaluation pipeline closes a gap in the main study itself. A classification system can flag public-green ticks as incomplete. An evaluator workbench can turn those flags into concrete, auditable engineering artifacts — a written diagnosis, a reproduction trace, an evidence quote, and where appropriate, a shadow patch. Together they make the trust gap inspectable rather than merely countable.

15. Threats to Validity

  • Benchmark author bias: Hidden tests, templates, and baseline suites were written by the same team, which may introduce systematic omissions.
  • Incomplete evaluator coverage: Only 10 of 101 false-green attempts have been audited by the evaluator model. 91 false-greens remain unreviewed.
  • Prompt privilege: Contract-visible prompts expose expected behavior, success signals and known failure modes. This is closer to a high-quality specification than ordinary bug-report context, and the improvement observed may not transfer to weaker contract descriptions.
  • Partial model coverage: Study A systems do not all cover every pack and lane combination. The Gemma supporting study uses separately configured local experiments not matched to Study A or B cells.
  • Local runtime confounding: Gemma experiments mix model capacity with local hardware, Ollama behavior, timeout policy and first-output boundaries. Public-red outcomes in those experiments do not isolate a single cause.
  • Scenario-family dependence: All 33 scenarios come from one authored benchmark suite. Generalizing to other task families, languages or domains requires additional evidence.
  • Hidden tests are operational definitions: They encode the authors' contract, not universal production truth. A passing hidden test means the patch satisfies those specific assertions.
  • Gateway confounding: North Mini product workflows change gateway and task pack together. Route effects and pack difficulty cannot be isolated.
  • Attempt independence: Separate runs lack memory, but model, provider and sampling configuration may not be fully controlled or observable.
  • Artifact provenance: Historical diffs may omit untracked files. Integrity hashes are computed after the run completes, not compared against real-time digests. Incomplete runs skipped by matrix resume are assumed complete.

16. Engineering Implications and Conclusion

This study began with a narrow question: can public CI green be trusted as a merge gate for coding-agent patches? The answer, consistently across model families, prompt lanes and scenario packs, is that it cannot be trusted alone. But the evidence accumulated over 391 depth-attempts, five breadth configurations and supporting local experiments supports a set of more specific and actionable findings.

Public green is not a merge gate for policy-heavy work. Across all six matched cells in Study B, the attempt-level false-green rate exceeded 24% under sparse prompts. On product workflows — where patches affect security, billing, inventory and SLA behavior — the pooled sparse false-green rate was 76.2%. Visible test success is a necessary checkpoint, not a sufficient one.

Contract exposure is a stronger lever than blind resampling. Exposing explicit acceptance criteria reduced the pooled false-green rate from 43.3% to 8.9%. Retries improved scenario coverage but did not eliminate false-greens; contract exposure did both. The decimal_money_rounding walkthrough demonstrates the mechanism: the agent's capability did not change; the information it received changed the search target from a local symptom to a policy.

Any-of-k coverage should not be confused with per-attempt trust. Laguna XS.2 reached 33/33 scenarios under any-of-three with contract-visible prompts — a strong coverage result. But 33/33 any-of-three success means at least one attempt succeeded per scenario, not that the system is reliable on any single invocation. Per-attempt hidden-pass rates remain the appropriate metric for deployment decisions that cannot wait for a retry.

Operational completion and semantic acceptance must be measured separately. The Gemma supporting study showed that the larger configuration reached public green more consistently, yet both sizes converged on the same false-green strategies. Separately, contract-visible batches in smaller configurations ended without reaching public green near first-output boundaries. A deployment pipeline must audit two distinct gates: did the system produce a viable patch, and does that patch satisfy the broader contract?

Evaluator judgments need deterministic evidence validation. Ten evaluator-agent reviews diagnosed authored-contract violations with evidence-backed verdicts. But the evaluator is itself a model, and its workbench is not yet a fully structured measurement instrument. Shadow-fix outcomes, reproduction status and inter-evaluator agreement remain open questions for the next iteration of the evaluation pipeline.

Deployment should be partitioned by task type, not governed by one global score. ci_forensics scenarios had false-green rates below 6% under contract-visible prompts for both systems. product_workflows remained at 32.3% for North Mini even with the contract exposed. A deployment strategy that treats all CI failures as equivalent will be simultaneously overconfident on mechanical tasks and underprepared for policy-critical ones.

The practical unit of evaluation is not the model. It is the deployed system, the information it receives, the acceptance boundary it must cross, and the evidence preserved when it fails. Measuring acceptance reliability requires measuring all four.
